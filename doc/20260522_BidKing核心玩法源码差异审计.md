# BidKing 核心玩法源码差异审计

时间：2026-05-22  
范围：对局流程、背包/道具、竞买人/英雄、藏品/仓库/珍阁、房间与地图门槛、结算与持久化。

## 一句话结论

当前实现已经接入了大量配置表、账号持久化、房间资金档位和部分珍阁/背包/竞买 AI 表现，但还不是源码级复刻。最大差异不在单个数值，而在底层模型：原版核心玩法围绕服务端权威的 `GameData`、`StockContainerData`、`StockBoxData`、`ItemData` 和逐回合日志运转；当前实现仍以自定义 `MatchRuntimeState`、扁平 `inventory`、扁平 `cabinetItemIds` 为主。因此“能玩”和“源码同构”之间还有一层必须补齐的核心数据结构。

这也解释了之前“余额 200 万但出价上限/开局资金不符合预期”这类问题为什么会漏：我们当时按现代 Demo 的房间资金和对局现金做了兼容实现，但没有把原版的“账号币、入场门槛、地图开局档位、服务端 `GameData` 日志、库存格子对象”一起落成同一套模型。

## 原版核心数据流

### 1. 入场与匹配

原版从 `BattlePrevPanel.EnterBattle()` 进入普通对局，关键步骤是：

- 必须选择竞买人/英雄。
- 检查当前 `BidMap` 的 `Costs` 和 `Requires`。
- 检查仓库空格是否低于 `ConstantConfig.grids_fullwarning_count`。
- 从当前战斗预设读取战斗道具，只有玩家实际拥有的道具才会进入 `useItems`。
- 若英雄皮肤选择变化，先同步皮肤。
- 调用 `Battle_Handler.C2S_MatchGame(mapId, heroId, items)`。

`C2S_MatchGame` 不是直接传道具 ID 给服务器，而是把每个道具 ID 转成 `SelectItemData(stockId, boxId)`。这个细节很重要：原版战斗道具也可以是库存格子里的实体物品，不是纯 refId 计数。

### 2. 服务端权威对局

原版客户端收到的是服务端下发的 `GameData`。`GameData` 包含：

- 当前地图、轮次、下一回合结束时间。
- `StockContainer`，即本局拍品容器。
- `UserLog`，每个玩家的出价日志、使用道具日志、是否弃权、英雄/皮肤等。
- `HeroSkillLog`、`MapSkillLog`、`ItemSkillLog`，技能效果由日志驱动。
- 本局可选/可用道具数量、携带上限、收益倍率等。

客户端的 `Battle_Handler.S2C_OnRoundStart(GameData)` 做的是回放和展示：

- 更新本地 `gameData`、`curRound`。
- 从 `UserLog.PriceLog` 取上回合出价。
- 派发 `E_OnRoundStart` 给 UI。
- 按地图技能、英雄技能、道具技能顺序播放日志。
- 进入本回合出价状态。

也就是说，原版对局表现不是客户端自由模拟，而是服务端日志驱动。

### 3. 出价、弃权与回合限制

`Battle_Handler.C2S_Auction(price)` 的核心逻辑：

- 加赛/超时回合要求本轮出价必须大于上一轮有效出价。
- 出价接近或超过账号币、超过地图提醒阈值、超过上一轮多倍时会二次确认。
- 普通模式走 `PlayerManager.GameBid(price)`。
- 训练模式走 `SimGameBidPrice(price)`。
- 私人房走 `RoomGameBid(price)`。

`Battle_Main.OnChujiaClick()` 里普通模式输入上限按 `PlayerGameData.Instance.coinNum`，也就是账号币，而不是地图开局资金档位。地图开局档位是房间/局内尺度，账号币是玩家真实余额与可承担风险。

### 4. 结算与入账

`S2C_OnGameOver` 收到 `S2C_45_game_over_notify`：

- 保存回放 `GamePlayBackData(GameData, UserSkillList, WinUserUid)`。
- 读取 `LossRecovery`。
- 记录收藏经验旧值/新值。
- 从 `GameData.StockContainer` 抽取最终赢得的物品。
- 进入结算 UI。

`Battle_Main.ShowTotalPrice(items, winnerCost)` 按 `Table_Item.SalePrice` 计算本局物品总价，收益为 `物品总价 - 成交价`。获胜后可以直接回收/出售结算物品，走 `PlayerManager.RecycleItems(stockId, boxId, count)`，服务端返回更新后的 `StockContainerData`。

这说明珍阁/仓库/藏品不是独立展示模块，而是对局胜利、入库、出售、收藏、收益入账的核心后半程。

### 5. 仓库、包裹与实体物品

原版有两类库存：

- `WarehouseItemData`：计数型包裹道具，字段是 `ItemCid`、`ItemCount`、`FirstGetTime`。
- `StockContainerData`：格子容器，包含 `StockBoxes`；每个 `StockBoxData` 里有 `BoxId`、`Position`、`ItemData`。

`ItemData` 不是一个简单 ID，至少包含：

- `Uid`
- `Cid`
- `Count`
- `BoxPositionData`
- `Rotate`
- `CanTrade`
- `No`
- `IsLock`
- `Quality`

`WareHouseData` 会根据 `Table_Item.slot_type` 和 `Size` 计算物品占格、旋转、摆放合法性。仓库支持移动、整理、排序、拆分、合并、装备/卸下仓库箱、锁定/解锁、回收。

当前我们只有扁平 `PlayerInventoryEntry { key, type, refId, quantity }`，无法表达上述实体物品生命周期。

### 6. 珍阁

原版珍阁也是 `StockContainerData` 的一种展示容器。`CabinetData.CheckItemCanSet()` 会检查：

- 物品是否允许入柜，`in_case` 和 `canSale` 都会参与。
- 单柜最大摆放数量 `place_max`。
- 柜子允许的 `location_type`。
- 物品占格面积不能超过 `max_slot_limit`。
- 物品品质必须满足 `quality_requirement`。
- 柜子格子与隔板占用不能冲突。

珍阁收益来自每个柜子的累计收益字段：

- `CabinetLastGetRewardTime`
- `CabinetCumulativeReward`
- `CabinetBasicReward`
- `CabinetReward`

领取奖励有冷却，使用 `ConstantConfig.cabinet_gaincoin`；柜子表还有 `coinbonus` 和 `timemax`。

当前实现的 `cabinetItemIds` 本质是“图鉴里挑几个 itemId 展示/产币”，不是原版“把实体库存物品移动进某个柜子的格子”。因此出售、入柜、收益和库存之间的关系还没有复刻到位。

### 7. 战斗道具

原版战斗道具选择来自 `BattlePrev_BattleSet` 预设，列表按 `Table_Item.ItemType.BattleItem` 和 `Table_BattleItem.battle_item_type` 过滤。进入对局时会把选中道具转换为 `SelectItemData(stockId, boxId)`。

局内使用道具：

- 普通模式走 `GameUseItem(itemId)`。
- 私人房走 `RoomUseItem(itemId)`。
- 训练模式走模拟用道具逻辑。
- 服务端返回 `ItemSkillLog`，客户端按日志播放技能效果。
- 普通模式每回合可用数量由 `CalRoundItemUseCount()` 控制，通常为 1。

当前实现是扁平 `equippedBattleItems: string[]`，使用时直接扣 `refId` 数量并生成本地效果，缺少实体库存位置、服务端技能日志、已使用标记和不同模式分支。

### 8. 竞买人/英雄

原版竞买人本质是 `HeroData`：

- 拥有的英雄由 `GetHeros()` 返回。
- `GetHeroState(heroId)` 区分永久拥有、限免、体验、未拥有。
- 皮肤通过包裹物品判断。
- 对局传入 `HeroCid`，皮肤传入 `HeroSkinCid`。
- 英雄技能来自 `Table_Hero.cast_type`，技能行为落到 `Table_Skill`、`SkillGroup`、`SkillEffect`。

当前实现大量地方仍使用 `roleId` 或按座位映射英雄，例如 AI 和技能取值用 `Hero[player.seat % Hero.length]`。这会导致“看起来有竞买人”，但不是真正的英雄拥有/选择/皮肤/技能链。

### 9. 地图、房间与开放门槛

原版地图入口来自 `Map` + `BidMap`：

- `Map.is_open` 控制开放。
- `open_time` 控制周期开启。
- `world_process` 控制世界进度门槛。
- `BidMap.Costs` 和 `Requires` 控制入场消耗和门槛。
- `daily_counts` 控制每日次数。
- `auction_limit_notify` 控制高价提醒。
- `bidder_number`、`Type` 等字段影响匹配类型和对局人数。

私人房有完整设置：

- 房间模式。
- 局数。
- 地图。
- 人数。
- 初始资金。
- 道具预算。
- 回合时间。
- 五档出价倍率。
- 可选英雄列表。
- 可选道具列表。
- 准备/取消准备/房主更新设置。

当前实现已经开始按余额筛房、按房间给开局资金，但地图开放时间、世界进度、每日次数、入场消耗扣除、完整私人房设置、可选英雄/道具列表仍缺。

| 出价上限 | 普通模式输入上限按账号币 `coinNum`，地图决定入场和局内尺度 | 已开始修成按房间资金档位，但核心出价仍是局内 cash 模型 | 需要明确账号币、开局资金、真实扣款之间的边界 | P0 |
| 地图入场消耗 | `BidMap.Costs` 校验且应进入消耗链 | 当前主要做 access 校验，门票有扣除，BidMap cost 未完整扣 | 玩家能进房但经济闭环不对 | P0 |

## 当前实现已经靠近源码的部分

- 已导入并使用大量原配置表，例如 `Item`、`BidMap`、`Map`、`BattleItem`、`Hero`、`Cabinet`、`RankAi`、`Constant`。
- 已建立账号、会话、密码哈希、访客升级、SQLite/JSON 存储。
- 已开始按 `BidMap`/房间选择决定开局资金，而不是简单按玩家余额。
- 已有地图房间、对局、结算、珍阁、包裹、竞买人、市场等前端入口。
- 已有 AI 行为树，并开始读取 `RankAi`。
- 已有背包出售、珍阁收益、战斗道具装备/使用等功能雏形。

这些是好底座，但它们目前更多是“功能等价雏形”，还没有完全切到原版的数据结构。

## 必须优先补的核心模型

### P0-1：服务端实体库存模型

新增源同构库存对象：

- `StockContainer`
- `StockBox`
- `InventoryItem`
- `WarehouseItem`
- `CabinetContainer`

至少要能持久化：

- stockId/cid
- boxId/position
- item uid/cid/count/rotate/quality/no/canTrade/isLock
- container 类型：仓库、珍阁、本局拍品、市场上架冻结等
- 包裹计数物品与格子物品分表或分域

没有这一层，珍阁、出售、战斗道具、市场、结算都只能继续做近似。

### P0-2：珍阁从 codex 展示改成实体容器

珍阁应改为：

- 每个柜子是一个 `StockContainerData`。
- 物品从仓库移动到柜子，而不是从图鉴添加。
- 入柜按 `CabinetData.CheckItemCanSet()` 复刻校验。
- 收益按柜子字段计算并持久化。
- 出售柜中物品时走实体 stock/box 回收，并同步移除柜子占格。

### P0-3：结算写入库存，而不是扁平奖励

对局结束应产生源形态结果：

- `GameData.StockContainer` 表示最终拍品实体。
- 胜者获得对应 `ItemData` 实体入库。
- 失败者应用 `LossRecovery`。
- 收藏经验按旧值/新值更新。
- 前端结算界面能直接回收本局实体物品。

### P0-4：对局日志源同构

服务端 match runtime 需要保存/输出：

- `GameData`
- `GameUserData`
- `GameUseItemOrPriceData`
- `GameSkillData`
- 每回合 `PriceLog`
- 每回合 `UseItemLog`
- 技能日志与目标索引
- game over replay payload

客户端逐步从本地自定义阶段展示改成消费这些日志。

### P0-5：账号与 profile API 绑定

正式体验里不能再让关键 profile 接口靠请求体的 `playerId` 决定操作对象。需要：

- 登录态/session 推导 profile。
- 管理/debug 接口单独隔离。
- profile 写操作全部服务端校验。
- 数据迁移与备份策略明确。

## 推荐开发顺序

1. 建 `StockContainer`/`StockBox`/`ItemData` 服务端模型和持久化迁移。
2. 把现有扁平库存迁移/兼容成“包裹计数物品 + 格子实体物品”双层库存。
3. 改结算：胜利奖励进入实体库存；出售走 `stockId + boxId`。
4. 改珍阁：从实体库存移动入柜，按柜子表校验、占格和收益。
5. 改战斗道具：携带时记录 `SelectItemData(stockId, boxId)`，局内使用产生日志。
6. 改英雄/竞买人：profile 保存 `HeroData` 和皮肤拥有状态，对局传真实 `HeroCid/SkinCid`。
7. 改房间/地图：补 `open_time`、`world_process`、`daily_counts`、入场消耗扣除、私人房完整设置。
8. 改 AI/技能：以真实 Hero 和服务端日志为中心接 `RankAi`、`Skill`、`SkillEffect`。
9. 改图鉴/收藏：从 `codex[]` 升级到 `CollectionItemList`、等级、经验、奖励领取。
10. 改市场：上架/交易/税费/锁定都围绕实体物品。

## 需要立刻纠偏的口径

- “珍阁是藏品背包”这句话在原版里更准确地说是：珍阁是实体藏品库存的一种展示/产币容器，物品仍然是带 stock/box/uid/quality/lock/trade 状态的实体。
- “玩家余额决定能进什么房”是对的；但“开局资金档位按房间/地图给”也是对的。两者不是一个字段。
- “出价上限”在普通模式源码里会参考账号币；但对局经济展示、AI 判断、结算收益又依赖本局地图/房间尺度。不能只用一个 `cash` 字段覆盖所有语义。
- “100% 复刻”的主线不应继续从 UI 表现推进，而应先把服务端权威数据和实体库存补上，否则每个玩法模块都会反复出现边界错位。
