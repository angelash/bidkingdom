## 源码版核心结论

- 原版局内核心是“多回合暗拍 + 信息披露 + 带入道具 + 竞价成交线”。玩家看不到完整仓库，只通过场景技能、竞买人技能、局内道具逐步获得轮廓、品质、价值、数量、格数等线索。
- 原版拍场由 `Map` 大场景和 `BidMap` 子拍场组成。`BidMap.auction_rounds_rate` 是每轮成交倍率，原始常见值为 `[2000,1600,1300,1100,0]`，可解释为 2.0 倍、1.6 倍、1.3 倍、1.1 倍、最终轮无倍率门槛。
- 原版共有 20 个竞买人。每个竞买人的实际局内技能由 `Hero.cast_type -> Skill.id -> SkillEffect.Category` 决定，面板描述只是这条链的本地化文案。
- 原版 `Item` 不是单纯“藏品表”。它覆盖货币、门票、仓库盒、收藏柜、角色、皮肤、体验卡、局内道具、核心藏品、特殊藏品等。真正进入拍场的藏品从 `BidMap.drop_group_id -> Drop -> Item` 生成。
- 原版 Unity 客户端是强服务端权威结构。真实对局通过协议接收 `GameData`，其中有 `StockContainer / UserLog / HeroSkillLog / MapSkillLog / ItemSkillLog / NextRoundTime`。客户端负责展示和交互，不在本地决定完整线上对局。
- `reverse/bidking/code/decompiled/Scripts/GameServerDemo` 是源码中附带的本地 Demo 服务器逻辑，用于还原部分规则，但它不是完整线上服务端。它能解释掉落、局部技能和成交线；角色技能、目标类型、效果分类以原始表和线上协议日志为准。
- 场景随机动画 `BattleRandom_Main` 和情报暗牌 `IntelligencePanel` 是源码 UI 行为证据。它们属于局内表现层流程，实际拍场和技能命中仍以服务端 `GameData` 为准。
- 房间、寄拍、邮件、商店、任务、错误码、敏感词、音频和引导都在源码/表中有独立维度。它们不是局内核心规则，但会影响完整复刻的入口、落账、提示和长期循环。

## 与 BidKingdom 复刻文档的边界

本组文档只说原版竞拍之王源码事实，例如“法蒂玛”“快递盲盒堆”“Battle_Main”“Table_BidMap”。如果需要看 BidKingdom 项目如何复刻和包装，应看 BidKingdom 实现策划归档目录。
