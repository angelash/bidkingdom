# 20260526 竞拍之王源码策划归档 04 - 原版技能与信息披露

## 技能表结构

`Skill` 表 256 行。核心字段含义：

| 字段 | 含义 |
| --- | --- |
| `id` | 技能 id，被 `Hero.cast_type`、`Item.skills`、`SkillGroup` 引用 |
| `skill_group` | 0 通常是角色/道具技能，非 0 用于技能组 |
| `skill_name` | 技能名本地化 key |
| `skilldesc` / `skill_textshow` | 技能描述或聚合线索文本 |
| `skill_type` | 技能类型，原表中常见 2 |
| `skilltarget / skilltargetvalue` | 第一目标筛选 |
| `skilltarget2 / skilltargetvalue2` | 第二目标筛选 |
| `skilltarget3 / skilltargetvalue3` | 第三目标筛选 |
| `skill_count_type / skill_count` | 数量模式和数量 |
| `skilleffect_position` | `SkillEffect.EffectId` 列表 |
| `skill_active_type` | 触发方式，角色/道具多为主动结算 |
| `skill_round / skill_CD` | 持续回合和冷却字段 |
| `show_type` | 展示分类 |

## SkillEffect 分类

`SkillEffect` 表 36 行，`Category` 是披露类型。原表有 1-28 类；源码本地 Demo 只实现了 1-9，但线上 `GameData` 会直接带技能结果，客户端按日志展示。

| Category | 策划含义 | 典型用途 |
| ---: | --- | --- |
| 1 | 显示轮廓/尺寸/位置 | 让玩家知道藏品占格和位置 |
| 2 | 统计总占格 | “所有 X 藏品总占位数为 N 格” |
| 3 | 统计平均占格 | “X 品质平均占位约 N 格” |
| 4 | 统计数量 | “本场共有 X 件” |
| 5 | 显示价格 | 单件价值揭示 |
| 6 | 显示藏品本体 | 完整揭示 itemId |
| 7 | 显示品质 | 只揭示品质 |
| 8 | 统计平均价值 | 某范围藏品平均价值 |
| 9 | 统计单位格价值 | 每格平均价值 |
| 10 | 统计总价 | 角色和道具的总价值情报 |
| 11 | 显示占格数 | 单件体量信息 |
| 12 | 品质文本 | 文本型线索 |
| 13 | 品类文本 | 文本型线索 |
| 14 | 价格位数 | 粗粒度价值提示 |
| 15-28 | 额外表驱动效果 | 原表保留，需按服务端日志、协议字段和原始配置推断 |

`SkillEffect.Category` 源表分布为：1-15 各 1 行；16、17、18、20、21、23、26、28 各 2 行；19、22、24、25、27 各 1 行。`EffectId` 是查表键，源码表类使用 `Table_SkillEffect.getByEffectId(effectid)`，不是按数组下标取值。

## 技能日志字段映射

技能效果最终都落到 `GameSkillData`，不同 `Category` 会读取不同字段：

| 线索类型 | 主要字段 | 展示语义 |
| --- | --- | --- |
| 单件/格子命中 | `HitItemIndex`、`HitBoxList` | 显示某件藏品、轮廓、占格、位置 |
| 品质/类型集合 | `HitItemQuilityList`、`HitItemTypeList` | 给出命中对象的品质或类型集合 |
| 总量统计 | `HitItemTotalPrice`、`TotalHitBoxIndex` | 显示总价值或总占格 |
| 平均统计 | `AllHitItemAvgPrice`、`AllHitBoxAvgPrice`、`AllHitItemAvgBoxIndex` | 显示平均价值、单格价值、平均占格 |
| 来源标识 | `SkillCid`、`HeroCid`、`MapCid`、`ItemCid` | 区分角色技能、拍场技能和局内道具技能 |

客户端按照这些日志字段播放情报，不应由 UI 重新扫描完整仓库推导未命中信息。

## 目标筛选口径

从 `GameServerDemo.Utils.GetItemsBySkillTarget()` 可直接确认的目标类型：

| target | 解释 | 示例 |
| ---: | --- | --- |
| 0 | 从全场随机取若干件 | 随机抽检、随机品质揭示 |
| 1 | 按 `Item.ItemType` 筛选 | 医疗用品、矿物珠宝、文玩古董 |
| 2 | 按 `item_quality` 筛选 | 白、绿、蓝、紫、金、红品质 |
| 3 | 按指定 `itemId` 筛选 | 指定藏品 |
| 4 | 从类型权重里抽一种类型，再筛选 | 伊森“随机 5 种类型” |
| 5 | 从品质权重里抽一种品质，再筛选 | 随机品质类技能 |
| 6 | 极值排序筛选 | 最高品质、最大占格、最高价值、单位格价值等 |

原表还出现 `skilltarget=7 / 10` 等高级口径。它们在本地 Demo 中没有完整实现，说明真实线上逻辑主要在服务端；Unity 客户端接收 `GameSkillData` 后做展示。

`Skill` 表存在三层目标：`skilltarget`、`skilltarget2`、`skilltarget3`。三层目标是叠加筛选条件，不是三选一。典型组合包括类型 + 品质、品质 + 极值、占格数 + 平均价值等。`skill_count_type / skill_count` 决定命中数量或比例，例如按固定数量、按比例或全量命中。

## 情报面板表现

源码中情报相关窗口分为两类：

| 窗口/类 | 用途 |
| --- | --- |
| `IntelligencePanel` / `IntelligenceItem` | 普通局首轮开局公共情报暗牌表现，单个情报项可加载 `SkillData.MapCid` 对应拍场图和名称 |
| `BattleIntelligence_Main` / `BattleIntelligence2_Main` / `BattleIntelligence_Main3` | 局内情报查看面板，按品质、类型、尺寸/格数等 Toggle 过滤仓库格 |

`Battle_Handler.S2C_OnRoundStart(GameData)` 在普通局首轮且存在 `MapSkillLog` 时调用 `OnOpenIntelligencePanel()`，等待 `TacketTask` 后才继续处理本轮 `MapSkillLog` 和 `HeroSkillLog`。源码硬条件是 `curRound == 0 && gameSkillDatas.Count > 0`，因此四张暗牌情报只在开局第一次出现，后续回合不再打开 `IntelligencePanel`。`IntelligencePanel.Build()` 初始化暗牌项，`PlayMaxTacketAni()` 中候选轮选时间约 `1f-1.9f`，停 `0.5s` 后明牌，明牌后再进入关闭动画。这说明情报动画是源码 UI 流程的一部分，但动画时长不改变技能命中内容。

局内中间情报列表只展示技能日志流，不把暗牌公共情报和角色私有线索再各自插入一次。协议层 `S2C_33_game_start_notify` / `S2C_37_game_next_round_notify` 下发的是包含 `MapSkillLog`、`HeroSkillLog`、`ItemSkillLog` 的整包 `GameData`，不是每条技能单独一个服务端推送；客户端拿到整包后在本地按日志逐条执行。原版 `Battle_Main.OnNewRound()` 不清空 `skillInfoRect.content`，`Battle_Main.AddSkillDesc()` 对 `show_type=0` 的技能先显示 `Battle_SkillShow` 技能卡，`MoveTo()` 完成后才实例化 `Battle_SkillDescItem` 到列表；`isShowingSkillDesc` 和 `caches` 队列保证技能日志逐条展示，已出现的列表项会累计保留。`Battle_Handler.DealSkillDataAsync()` 会等待技能展示任务释放后才继续下一条并同步仓库变化，因此仓库格信息不应先于对应技能提示和列表入列一次性出现。当前普通局可见流程按首轮 `MapSkillLog` 入列一次场地情报，第二轮起只追加角色和道具触发信息。

## 信息披露层级

原版信息披露不是“直接知道整仓”，而是分层推进：

| 层级 | 玩家得到什么 | 对竞价的作用 |
| --- | --- | --- |
| 数量 | 某类/某品质共有多少件 | 判断类型池和稀有度 |
| 轮廓 | 位置、形状、占格 | 推测仓库布局和大件价值 |
| 品质 | 白到红的品质等级 | 估算价值上限 |
| 价值 | 单件或聚合价值 | 直接影响出价 |
| 完整藏品 | `itemId` 本体 | 明确价值、类型、收藏属性 |
| 聚合统计 | 总价、总格数、平均格数、单位格价值 | 宏观判断本局值不值得追 |

## 场景技能组

`SkillGroup` 只有 5 行，分别对应第一至第五回合。每组里是 `Skill.id + 权重`：

| groupid | 回合 | 用途 |
| ---: | --- | --- |
| 101 | 第一回合 | 开局给基础场景信息 |
| 102 | 第二回合 | 补充中期场景信息 |
| 103 | 第三回合 | 常用随机场景技能池 |
| 104 | 第四回合 | 后期信息补偿 |
| 105 | 第五回合 | 最终轮少量场景信息 |

`BidMap.map_random_skill` 在源表中保存按回合的场景技能组线索。普通局可见场地情报只按首轮开局一次验收；第二轮起不再把 `map_random_skill` 主动转成新的场地情报条目，后续信息依赖角色和道具触发。

## 设计结论

原版技能系统的核心不是伤害或 Buff，而是“信息不完全条件下的估值能力”。角色、场景、道具都只是不同的信息渠道：

- 角色定义玩家长期策略偏好。
- 场景技能给所有人一些公共信息。
- 局内道具消耗资源换取临时私有信息。
- 技能日志只暴露命中的结果，不应把未命中的仓库真相泄漏出来。
