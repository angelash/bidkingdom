## 关键差异

| 主题 | 原版源码事实 | 当前工程/旧 Demo 注意点 |
| --- | --- | --- |
| 场景数量 | `BidMap` 125 行，含普通、训练、主题变体 | 当前工程可能只挑一部分作为可玩拍场 |
| 角色 | 原版 20 个国际化人物，法蒂玛、陈美、艾莎等 | 当前工程可改包装名和立绘，但不可改技能链含义 |
| 局内状态 | 原版客户端接收 `GameData`，含三类技能日志 | 旧 Demo 在本地计算部分技能，真实线上应服务端权威 |
| 角色技能触发 | `Hero.cast_type` 可跨回合配置，非 0 指向 `Skill.id` | 旧 Demo `EnterRound()` 只在第 1 回合处理玩家技能，是简化逻辑 |
| 技能目标 | 原表有 `skilltarget / skilltarget2 / skilltarget3` 和更多目标类型 | 旧 Demo 只完整实现 0-6 部分口径 |
| SkillEffect | 原表有 36 行、Category 1-28 | 旧 Demo 只处理 1-9 |
| 拍场随机 | `BidMap.map_group` 非空时显示随机拍场动画 | 当前工程如直接选定 BidMap，则少了原版随机揭示仪式 |
| 成交线 | 常规 `[2000,1600,1300,1100,0]`，按千分比解释 | 当前实现需确认是否按 `max > second * rate` 或等价规则 |
| 道具 | 64 个 `BattleItem`，全部走 `Item.skills` | 当前工程需保证 itemName 直连技能，而不是自创道具效果 |
| 藏品 | `Item` 1136 行，核心藏品是其中一部分，支持多类型标签 | 不应把 `Item.is_auction=0` 误读为“不可进入竞拍”，拍场由 `Drop` 生成 |
| 局外 | UIWnd 80 个窗口，表覆盖任务、商店、排行、协会、邮件、活动 | 当前工程 Web 化后结构不同，但数据口径应能回溯原表 |

## 后续复核建议

- 若要做 100% 原版还原，应优先对照 `Hero.cast_type`、`Skill`、`SkillEffect`、`BattleItem`、`BidMap.drop_group_id` 和 `RankAi`。
- 本项目已明确采用“玩法 100% 原版还原 + 三国表现包装”口径：角色名、剧情、视觉、音频和 UI 皮肤可替换，但数值、流程、技能目标、效果、信息披露层级、掉落、AI 和结算不可改造。
- 旧 Demo 可用于理解表字段意图，但不要把它当作完整线上服务端规则。
