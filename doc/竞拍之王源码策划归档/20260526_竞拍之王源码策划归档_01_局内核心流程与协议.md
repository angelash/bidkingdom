# 20260526 竞拍之王源码策划归档 01 - 局内核心流程与协议

## 局内主循环

原版源码的局内流程可以归纳为：

```mermaid
flowchart TD
  A["战前选择 BidMap / Hero / BattleItem"] --> B["校验门票、资产门槛、仓库容量"]
  B --> C["C2S_28_match_game"]
  C --> D["匹配成功 S2C_29 / S2C_OnMatchGameSuc"]
  D --> E["GameData 初始化 StockContainer / UserLog"]
  E --> F["回合开始 S2C_37_game_next_round_notify 或 GameData 更新"]
  F --> G["展示上一轮出价和玩家状态"]
  G --> H["执行 MapSkillLog"]
  H --> I["执行 HeroSkillLog"]
  I --> J["玩家可使用一次道具或直接出价"]
  J --> K["C2S_38_game_use_item / C2S_34_game_bid / C2S_42_game_stand_down"]
  K --> L{"成交线判定"}
  L -->|未成交| F
  L -->|成交或最终轮| M["S2C_45_game_over_notify"]
  M --> N["最终揭示仓库、赢家、成本、净收益"]
```

## 战前入口

战前 UI 在 `BattlePrevPanel_Main / BattlePrevPanel / BattlePrev_BattleSet / BattlePrev_MapItemPanel` 一组类中完成：

- `BattlePrev_MapItemPanel` 展示 `BidMap` 名称、描述、资源图、费用、资产要求、回合数量和人数。
- `BattlePrevPanel.EnterBattle()` 先校验 `curBidMap.Costs.CheckItemEnough()` 和 `curBidMap.Requires.CheckItemEnough()`。
- 如果金币不足，会进入低保检查：当前资产低于 `relief_fund_limit` 且今日次数未满时，引导领取低保。
- 如果仓库格满，提示 `ui_gridsfull_notify`。
- 通过校验后保存角色、皮肤、带入道具预设，并调用 `Battle_Handler.C2S_MatchGame(curBidMap.id, heroId, useItems, callback)`。

## 匹配和开局协议

`PlayerManager.MatchGame()` 发送：

| 协议 | 字段 | 含义 |
| --- | --- | --- |
| `C2S_28_match_game` | `Token` | 玩家登录令牌 |
|  | `MapCid` | 选择的 `BidMap.id` |
|  | `HeroCid` | 选择的 `Hero.id` |
|  | `MatchType` | 匹配类型 |
|  | `SelectItemList` | 带入局内道具 |

匹配成功后，客户端进入 `Match_Main`，再由服务器推送真实 `GameData`。原版客户端还保留 `GameServerDemo.ServerHandler`，它会在本地按 `bidder_number` 组房间，但这只是 Demo 逻辑。

## GameData 数据骨架

`Protodata/GameData.cs` 是局内权威快照。关键字段：

| 字段 | 含义 |
| --- | --- |
| `Uid` | 对局唯一 id |
| `MapId` | 本局实际 `BidMap.id` |
| `Round` | 当前轮，客户端展示时加 1 |
| `StockContainer` | 本局仓库与藏品格数据 |
| `UserLog` | 玩家、角色、皮肤、出价、弃权、用道具记录 |
| `HeroSkillLog` | 角色技能触发日志 |
| `MapSkillLog` | 场景技能触发日志 |
| `ItemSkillLog` | 局内道具触发日志 |
| `NextRoundTime` | 下一轮截止时间，客户端按 Unix 秒转本地时间 |
| `SelectItemCount` | 可选道具数量 |
| `RoundCanUseItemCount` | 每轮可用道具次数 |
| `GameCarryItemMax` | 带入上限 |
| `GameGoldRateMax` | 金币倍率/限制上限 |
| `GameType` | 普通、暗拍、房间等类型 |
| `ServerTime` | 服务端时间同步，客户端 `SyncTime()` 中乘 1000 写入毫秒计时器 |

`GameData` 中的三个技能日志列表都使用 `GameSkillData`。客户端不重新计算技能命中，只按服务端日志展示结果。

| `GameSkillData` 字段 | 作用 |
| --- | --- |
| `SkillCid` | 命中的 `Skill.id` |
| `HeroCid` / `MapCid` / `ItemCid` | 来源角色、来源拍场或来源局内道具 |
| `CastTime` / `CastRound` | 触发时间和触发回合 |
| `HitItemIndex` | 命中藏品索引 |
| `HitBoxList` | 命中格子列表，元素为 `BoxInfoData` |
| `AllHitItemAvgPrice` / `AllHitBoxAvgPrice` / `AllHitItemAvgBoxIndex` | 平均价值、每格平均价值、平均占格类聚合线索 |
| `HitItemTotalPrice` / `TotalHitBoxIndex` | 总价值、总占格类聚合线索 |
| `HitItemTypeList` / `HitItemQuilityList` | 命中类型列表和品质列表 |

`UserLog` 使用 `GameUserData`。它同时承担玩家身份、出价、弃权和带入道具记录：

| `GameUserData` 字段 | 作用 |
| --- | --- |
| `UserUid` / `Name` | 玩家唯一标识和显示名 |
| `HeroCid` / `HeroSkinCid` | 局内竞买人和皮肤 |
| `UseItemLog` | 本玩家用过的局内道具记录 |
| `PriceLog` | 每轮出价记录 |
| `IsStandDown` / `IsQuit` | 弃权/离开状态 |
| `HeadCid` / `HeadBoxCid` / `TitleCid` | 头像、头像框、称号 |
| `SimSelectItemList` / `SimBuffItemList` | 训练局专用道具和 Buff |
| `SelectItemList` | 普通局/房间带入的局内道具 |

结算协议 `S2C_45_game_over_notify` 继续携带 `GameData`，并补充 `WinUserUid`、`OldCollectionExp`、`NewCollectionExp`、`LossRecovery` 和 `UserSkillList`。`UserSkillList` 的元素是 `GameUserSkillLogData`，按玩家归档 `HeroSkillLog / MapSkillLog / ItemSkillLog`，用于结算页和回放复原技能贡献。

## 终局仓库揭露表现

终局协议到达后，客户端由 `Battle_Main.DoItemEffect(List<GridItemData> itemDatas, int winnerCost, PlayerInBattleData winnerData, List<PlayerInBattleData> players)` 驱动最终仓库揭露：

1. 先隐藏情报面板，展示 `endPrevPanel` 得主预览。
2. 激活 `wareHouse` 和 `resultPanel`。
3. 遍历 `itemDatas`，获取或创建 `GridItem`，调用 `GridItem.Build(data)` 写入真实藏品数据，再立即调用 `GridItem.DoSearchLoop()`。
4. `GridItem.DoSearchLoop()` 只播放 `ani_search_loop`，并把 `canShowDetail=false`，因此格子进入 loading 搜索态，品质、名称、图标和价格不作为最终揭露内容呈现。
5. `items.Sort((a, b) => a.itemData.pos - b.itemData.pos)` 后按仓库 `pos` 从小到大逐个揭露。
6. 每件藏品调用 `await GridItem.DoEffect(speed)`；动画结束后 `canShowDetail=true`，该件藏品的品质、图标、名称和价值进入可见状态，并加入 `showItems` 后刷新总价。
7. 全部揭露完成后 `bQuickEnd=true`，再显示退出或领取按钮。

得主预览是最终仓库揭露之前的独立步骤。`Battle_Handler.S2C_OnGameOver(S2C_45_game_over_notify / S2C_207_room_game_over_notify)` 先 `OnPlayerAuctionNotify(..., isGameOver: true)` 同步最终出价，再 `S2C_OnFinalDataSync(playerDatas)`，随后 `await TaskUtil.Delay(1000)`，派发 `E_OnGameOver`。`Battle_Main.DoItemEffect()` 在 `winnerData != null` 时隐藏 `BattleIntelligence2_Main / BattleIntelligence_Main`，执行 `await endPrevPanel.Show(heroId, skinId, winnerName, isWin, itemIds, datas)`；`Battle_EndPrevPanel.Show()` 写入 `"ui_bidwin"` 或 `"ui_bidfail"`（两者文案均为“最高出价！”、颜色/背景不同）、得主头像和 `winnerNameTxt`，并把 `EndPrevPlayerData` 的最终轮出价列表铺到 `playerTrans`，然后 `DOVirtual.DelayedCall(3f, SetActive(false))`。只有该预览隐藏、`Task` 结束后，`DoItemEffect()` 才激活 `wareHouse` 与 `resultPanel`，进入逐件 loading/揭露。

`GridItem.DoEffect(int speed=1)` 的品质动画固定如下：

| `GridItemData.quality` | 动画 | 原始 clip 时长 | 默认等待 |
| ---: | --- | ---: | ---: |
| 1 | `ani_bai` | 0.7 s | 1000 ms |
| 2 | `ani_lv` | 0.7 s | 1000 ms |
| 3 | `ani_lan` | 1.1 s | 1000 ms |
| 4 | `ani_zi` | 1.633 s | 2000 ms |
| 5 | `ani_jin` | 2.1 s | 2000 ms |
| 6 | `ani_hong` | 2.567 s | 3000 ms |
| 7 | `ani_caise` | 3.583 s | 4000 ms |

等待公式来自源码：`time = animator.PlayAnim(animationName); time += 0.5f; await delayTask.Delay((int)time * 1000 / speed);`。`MainUtils.PlayAnim(Animator animator, string clipName)` 从 `runtimeAnimatorController.animationClips` 查找同名 clip，播放并返回 `clip.length`。`ani_search_loop` 的 clip 时长约 1.333 s 且 `LoopTime=true`，loading 图素为 `ui/prefab/uimain/griditem.data` 中的 `img_loading_1`。私人房间揭露速度强制 `speed=2`，其余模式读取 `PlayerGameData.Instance.userSetting.LocalSaveData.int_setting_openspeed_save`。

## 局内仓库面板

`Battle_Main.Init()` 创建局内仓库时只执行 `WareHouseData.Build(40)`、`wareHouse.SetData(houseData)` 和 `wareHouse.InitGrids()`，其中宽度固定为 10，行数为 40。`WareHouse.InitGrids()` 逐格实例化 `Grid` prefab，`Grid.Set(WareHouse)` 会启用格子 Image；该格子素材来自 `ui/prefab/uimain/grid.data` 的 `itembox_pic_1`。局内情报同步只通过 `OnBattleItemSync(E_OnBattleItemSync evt)` 调用 `wareHouse.data.AddItem(evt.gridDatas[i])`，随后 `CalYuguPrice()` 刷新 `yuguTxt`。

局内仓库面板本体不承载“最终竞拍价格 / 已开珍物估值 / 当前盈余”等结算摘要；但局内 HUD 保留 `yuguTxt` 当前预估最低价格。`CalYuguPrice()` 遍历 `wareHouse.data.battleGridItemDatas`，对每条 `BattleGridItemData.GetYujiPrice().x` 求和并写入 `"ui_battle_yuji".GetLangStr(minPrice.ToString("N0"))`，`yugu_imgBg` 随文本非空显示。`GetYujiPrice()` 的过滤粒度为：已有 `price` 直接返回确定价；已有 `itemId` 按具体 `Table_Item` 取价格；否则调用 `Table_Item.GetItemsBuyAttributes(Size, sizeCount, rank)`，只按轮廓尺寸、占格数和品质取候选最低/最高价，`itemType` 不参与估价过滤。

未知格、轮廓、品质、价格和占格文本都由 `GridItem.Build(BattleGridItemData)` 写入正式 `GridItem` prefab：`rankImg.LoadRankImg(gridItemData.rank)` 设置品质框，`iconImg` 只在真实 `itemId` 已知时显示，`lunkuoGo` 只在 `gridItemData.lunkuo` 为真时显示，`price` 和 `sizeCount` 分别写入 `nameTxt` 与 `numTxt`。

## 回合开始处理

`Battle_Handler.S2C_OnRoundStart(GameData)` 做以下事：

1. 缓存 `gameData`，读取 `MapId`、`Round`、`MapSkillLog`。
2. 从 `MapSkillLog` 中筛出 `CastRound == roundNum` 的场景技能。
3. 从 `HeroSkillLog` 中筛出当前回合角色技能。
4. 用 `UserLog` 构建 `PlayerInBattleData`，并同步自己是否已弃权。
5. 如果不是第 1 个回合，展示上一轮所有玩家的出价。
6. 触发 `E_OnRoundStart`，由 `Battle_Main` 刷新玩家栏、回合、倒计时、局内 UI。
7. 依次处理场景技能和角色技能，展示技能描述和同步已披露的格子。
8. 派发 `E_OnRoundStartBid`，玩家可以开始操作。

普通对局在第 0 轮如存在场景情报，会自动打开 `BattleIntelligence` 面板；训练、回放、重连会走各自的缓存和补放逻辑。

## 玩家回合操作

玩家每轮核心操作有三类：

| 操作 | 客户端方法 | 协议 | 限制 |
| --- | --- | --- | --- |
| 出价 | `Battle_Handler.C2S_Auction(price)` | `C2S_34_game_bid` | 已弃权、已出价、回放状态不可出价 |
| 使用道具 | `Battle_Handler.C2S_UseItem(data)` | `C2S_38_game_use_item` | `UseItemLog` 当前轮没有记录时才可用 |
| 弃权 | `Battle_Handler.C2S_Qiquan()` | `C2S_42_game_stand_down` | 弃权后不可再出价/用道具 |

`Battle_InputDevice` 提供出价输入、隐藏金额、金额颜色、金额上限和快捷位输入。金额颜色阈值是 `<10000` 灰、`<100000` 绿、`<1000000` 蓝、`<10000000` 橙、以上红。

## 成交线

原版 `BidMap.auction_rounds_rate` 是每轮成交线。源码附带的 `GameServerDemo.CheckGameOver()` 逻辑是：

- 取本轮最高价 `maxPrice` 和第二高价 `secondMax`。
- 当前轮倍率 `rate = auction_rounds_rate[round - 1] * 0.001`。
- 若 `maxPrice > secondMax * rate`，本轮成交。
- 若没有达到成交线，进入下一轮。
- 常见倍率 `[2000,1600,1300,1100,0]` 对应前四轮逐渐降低门槛，第五轮最终成交。

真实线上服务端不会由 Unity 客户端本地判定，它通过 `S2C_45_game_over_notify` 推送最终结果；但上述 Demo 逻辑清楚说明了表字段的设计意图。

## 局内模式

| 模式 | 源码入口 | 特点 |
| --- | --- | --- |
| 普通匹配 | `BattleType.Normal` | 通过 `C2S_28_match_game` 匹配，服务端推送 `GameData` |
| 模拟训练 | `BattleType.Training` | `CreateSimGame` 创建，使用 `SimSelectItemList / SimBuffItemList` |
| 私人房间 | `BattleType.PrivateRoom` | 使用 Room 系列协议，房主配置人数、时间、倍率、道具预算 |
| 回放 | `BattleType.Playback` | 从历史 `GamePlayBackData` 逐轮播放技能和出价 |
| 重连 | `S2C_OnBattleReconnect` | 按 `GameData` 中所有日志补放到当前轮 |

普通匹配不是私人房间流程。`BattlePrevPanel.EnterBattle()` 在入场条件、资产、格子容量和带入道具检查通过后隐藏战前地图/底栏，调用 `Battle_Handler.C2S_MatchGame(curBidMap.id, battleSet.heroId, useItems, OnMatchCallback)`。`Battle_Handler.C2S_MatchGame()` 设置 `battleType = BattleType.Normal`，组装 `C2S_28_match_game`，字段包括 `MapCid`、`HeroCid`、`MatchType` 和 `SelectItemList`。`PlayerManager.MatchGame()` 收到 `S2C_29_match_game` 后把 `EstimatedMatchTime` 写入 `Battle_Handler.matchTime`，回调显示 `Match_Main`。

`Match_Main.OnShowImp()` 从 00:00 开始计时，显示 `yugutime` 预计匹配时间，并监听 `E_OnMatchCallback`。服务端匹配成功后 `Battle_Handler.S2C_OnMatchGameSuc()` 派发成功事件，`Match_Main.OnMatchCallback()` 隐藏匹配面板，回退到 `BattlePrevPanel_Main` 或 `UIMain`，随机拍场先显示 `BattleRandom_Main`，随后显示 `Battle_Main`。取消匹配走 `C2S_30_cancel_match_game / S2C_31_cancel_match_game`，不进入 `PrivateRoom_Room`。

视频 `BidKing 2026-05-18 10-42-08` 的 30s-60s 与源码一致：30s-52s 停留在 `BattlePrevPanel` 战前选择，点击开始行动后约 53s 关闭场景详情并回到大场景地图，地图上显示 `Match_Main` 式匹配悬浮窗，包含 00:00 计时、预计匹配时间和取消入口；55s-58s 计时递增，匹配成功后进入随机拍场 / 战斗流程。该段未出现 `PrivateRoom_Room` 的等待、准备或房主开始界面。

## 房间协议骨架

房间模式的权威数据是 `RoomData`，核心字段如下：

| 字段 | 策划含义 |
| --- | --- |
| `RoomUid` / `OwnerUid` / `RoomName` | 房间 id、房主和房名 |
| `RoomMode` | 房间竞拍模式 |
| `BigGameCount` | 房间连续竞拍场次 |
| `MapCid` | 房间选择的拍场 |
| `InitialBudget` / `ItemBudget` | 初始预算和局内道具预算 |
| `RoundTime` | 每轮竞价时间 |
| `RoomPlayernum` | 房间人数 |
| `CanSelectHeros` / `CanSelectItems` | 可选竞买人和可选局内道具 |
| `BidRate` | 自定义成交倍率 |
| `UserList` / `ReadyUserList` / `RoomReadyUserList` | 房间玩家和准备状态 |
| `GameData` | 房间局内权威快照 |

Room 协议覆盖创建、邀请、加入、退出、准备、开局、用道具、出价、下一轮、弃权、表情、结算、更新设置、转让房主、解散、搜索和踢人。房间局内仍复用 `GameData` 和 `GameSkillData`，只是在协议号和外层 `RoomUid` 上区分普通匹配。
