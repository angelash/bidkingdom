# BidKing 异常状态管理需求分析与实现记录

日期：2026-05-21

## 原游戏设计线索

- 原表体系已有 `ErrorCode / Notice / UIWnd / Guide / Sound`：错误码负责业务失败文案，Notice/UIWnd 负责系统弹窗与窗口层级，Guide 负责流程引导。
- 现有复刻已把 HTTP 4xx/5xx 包装成 ErrorCode 信封，但 socket 局内错误、重连失败、连接断开和阶段卡死仍分散在 toast 或静默状态清理中。
- 原游戏处理异常的核心体验不是只给一行文字，而是：明确阻断原因、保留最近提示、给出可执行动作，例如返回主界面、重新同步、重新登录或刷新。

## 异常分类

| 分类 | 触发场景 | UI 表达 | 默认处理 |
| --- | --- | --- | --- |
| 连接异常 | websocket 断开、重连中 | 右下异常入口 + toast | 重新同步 |
| 房间异常 | 房间不存在、席位不存在、dev 重启丢内存房间 | 阻断弹窗 | 返回主界面 |
| 对局异常 | 阶段计时结束后长期未推进 | 阻断/警告弹窗 | 请求快照同步 |
| 档案异常 | 背包、商城、任务、协会等 profile 操作失败 | 异常汇总 + toast | 知道了 |
| 账号异常 | 账号会话切换、会话失效 | 阻断弹窗 | 返回主界面或重新登录 |
| 服务端业务异常 | socket 操作抛错、HTTP ErrorCode | 按 ErrorCode 分级 | 由异常中心承接 |

## 本轮实现

- 新增前端异常状态模型：`gameExceptionRuntime.ts`。
- 新增异常状态中心 hook：`useGameExceptionCenter.ts`，统一保存活跃/最近异常、去重计数、处理状态。
- 新增异常弹窗与汇总入口：`GameExceptionCenter.tsx`。
- `BidKingApp` 接入异常中心，统一承接 profile 操作失败、socket 错误、房间重连失败、连接断开和局内阶段超时。
- `useBidKingSocket` 新增 `onException` 回调，不再只依赖 toast。
- 服务端 socket `emitError` 复用 `apiErrorEnvelope`，让局内业务错误也带 `CODE_x` 风格文案。

## 后续扩展点

- 将 HTTP `fetchProfileSnapshot`、后台管理、市场订单拉取等非 action 请求也包装为同一异常输入。
- 将服务端 `toast` 事件扩展为结构化异常事件，直接携带 `errorCodeId / action / recoverable`。
- 对局阶段卡死可进一步接后台审计：自动附带 roomCode、matchId、roundId 并写入 admin 事件流。
