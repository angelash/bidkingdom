# 20260525 BidKing 竞买人改版对应与当前实现核对归档

> 目的：在原版事实表之上，固定本项目改版的一一对应关系，并列出当前实现需要修改的点。本文档只归档方案，不直接改变运行代码。

## 不可变原则

- 顺序不可变：竞买人面板、战前选择、局内选人、解锁列表都应使用截图顺序 `104,107,108,101,105,110,208,209,102,103,106,109,201,202,203,204,205,206,207,301`。
- 功能不可变：每个包装角色必须绑定固定 `sourceHeroId`，局内效果继续读取该 Hero 的 `cast_type`、`Skill` 和 `SkillEffect`。
- 文案可改：姓名、称号、背景可以做 clean-room 题材包装，但要能解释同一技能效果，不能把“品质揭示”写成“价值估算”，也不能把“第 5 回合触发”写成“每回合触发”。
- 外观可改：立绘、头像、色调可以替换，但不能影响 `HeroSkin.skinhero`、语音、BGM 和解锁状态的源表引用。

## 只改配置是否足够

结论：不够。只改包装配置可以解决“名字、背景、详情文案、立绘对应”的一部分，但不能保证技能效果一一等价。

当前已经够用的部分：

- `compatRuntime.skillForHero()` 已按 `Hero.cast_type[roundIndex]` 读取技能，能支撑每回合自动触发和第 5 回合才触发这类节奏。
- `skillTargeting` 已能解释随机、品类、品质、排序、已知/未知状态等主要目标筛选。
- `skillEffectRuntime` 已能把轮廓、品质、价格、数量、总价、平均占格等主要 `Category` 映射成仓库知识或情报文本。

当前还必须改代码/运行解释的部分：

- 竞买人顺序和绑定必须从“数组下标”改成显式 `sourceHeroId`，否则配置文案排对了，局内技能仍可能绑错。
- `skill_count_type=2` 的百分比计数尚未按比例解释，吴起灵第 4 步会偏离原版。
- 多效果技能的 UI 文案和命中摘要需要读取完整 `skilleffect_position`，不能只按第一个效果描述。
- 部分原版面板效果是复合效果，但当前 `Skill` 行只覆盖其中一段，典型包括 `110/稀世估值`、`106/灵魂和声`、`203/食珍双鉴`、`206/一期一现`。这些需要新增复合技能解释或逐项确认原表是否另有隐藏触发来源，不能只靠改显示文案糊过去。

## 推荐实施顺序

1. 建立 `sourceHeroId` 为主键的竞买人 catalog，内含截图顺序、包装文案、立绘 roleId、原表证据引用。
2. 前端 `BidderPanelView`、战前选择、房间角色选择统一使用 catalog 顺序渲染。
3. 服务端与 profile 选择逻辑从 roleId 下标映射改成显式 `sourceHeroId` 校验。
4. 替换 `roleSkillDetails` 为按 `sourceHeroId` 的 20 份技能说明，文案必须从 `Hero.hero_effect_description + cast_type Skill` 提炼。
5. 补 `skill_count_type=2` 运行解释，并补多效果技能的信息流文案。
6. 做一轮自动化快照：竞买人面板 20 个顺序、局内 5 回合技能日志、典型角色 `104/107/207/301` 的触发时机。
