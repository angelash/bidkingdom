# 珍宝局核心复刻包装化修改计划

## 目标结论

下一轮开发不再扩展市场、商城、成长、活动等周边系统，而是把当前 Demo 收敛成“珍宝局包装下的 BidKing 核心复刻”：

```text
拍卖大厅匹配
→ 随机藏宝仓滚动抽选
→ 展示仓型包装与风险先验
→ 拍卖师随机情报
→ 同一藏宝仓 5 轮渐进揭示
→ 明拍或暗拍出价
→ 每轮排名/差距反馈
→ 角色技能按轮提供非对称情报
→ 第 5 轮或提前满足成交线后成交
→ 原地逐格开箱结算
```

本轮修改的核心判断标准：玩家必须能围绕同一个仓库，在 5 轮内持续用“仓库格信息、拍卖师情报、角色技能、对手报价轨迹”修正自己的心理价。

## 包装口径

BidKing 的“仓库、集装箱、废弃仓、古董工艺箱”只作为机制参考。我们这边继续使用《珍宝局》的三国/古藏包装，但包装不得改变核心玩法。

| BidKing 机制 | 珍宝局包装 | 实现要求 |
| --- | --- | --- |
| 竞拍大厅地图 | 珍宝局拍卖大厅 / 各地藏宝仓节点 | 先做可用大厅，不做复杂世界地图 |
| 仓库/集装箱 | 藏宝仓 / 封存柜 / 商船仓 / 军库 / 旧宫库 | 仓型只影响背景、藏品池、风险文案、数值分布 |
| 战利品格 | 珍宝格 / 藏品格 | 仍然是右侧空间化格子主板 |
| 拍卖师情报 | 掌眼人 / 拍卖师 / 主持人情报 | 第 1 轮前必须抽一条公共情报 |
| 角色技能 | 掌柜/鉴宝师/行商/谋士技能 | 技能只改变信息视野，不直接加钱 |
| 开箱结算 | 开仓验宝 | 藏品在原格逐件揭开，收益动态变化 |

当前配置里的 `洛阳旧宫`、`江东商船`、`西凉军库`、`黑市密仓`、`古战场遗址`、`破败书院` 可以继续作为首批仓型包装；但要从“5 个样例轮次”改成“单仓候选模板”。

## 当前实现判断

当前工程已经有可复用基础：

- 固定端口与启动脚本已完成。
- 4 人房间、明拍/暗拍选择、AI 补位已完成。
- `coreMode` 已经支持同仓 5 轮的基本结构。
- 出价键盘、二次确认、左侧 5 轮排名、原地开箱已有基础。
- 服务端权威结算、明拍公开金额、暗拍隐藏金额已有基础。

但距离最新素材确认的核心体验仍有以下差距：

| 模块 | 当前状态 | 必须修改 |
| --- | --- | --- |
| 随机仓 | 有表现层抽选 | 需要接入真实仓型候选、结果卡、仓型文案和局前流程 |
| 拍卖师情报 | 文本规则化生成 | 改成第 1 轮前抽情报卡，情报直接改变仓库格或统计信息 |
| 角色技能 | 多数仍是文本线索 | 改成按轮触发的信息视野更新，能影响右侧仓库格 |
| 仓库格知识 | 主要按轮全局揭示 | 改成公共知识 + 私人知识 + 技能来源叠加 |
| 实时情报 | 基本缺失 | 增加按品质/类型/轮廓筛选候选藏品的估价工具 |
| AI 估价 | 基于公共估值和文本线索 | 改成基于 AI 自己可见知识推断，不读取真实完整数据 |
| 数据配置 | `scriptedRounds` 仍像旧 5 轮样例 | 改成单仓模板、藏品池、拍卖师情报牌、角色技能配置 |
| UI 流程 | 对局主流程已通 | 需要补齐“匹配 → 随机仓 → 情报卡 → 第 1 轮”的完整节奏 |

### 2. 对局状态机

修改文件范围：

- `packages/match-core/src/types.ts`
- `packages/match-core/src/match.ts`
- `apps/server/src/roomManager.ts`
- `packages/shared/src/index.ts`

当前 `startNextRound` 在 `coreMode` 下直接进入 `auction`，需要增加开局阶段。

目标状态机：

```text
WAITING_ROOM
→ MATCHING
→ WAREHOUSE_ROLLING
→ WAREHOUSE_SELECTED
→ AUCTIONEER_REVEAL
→ ROUND_INTEL
→ AUCTION
→ ROUND_FEEDBACK
→ 下一轮 ROUND_INTEL / AUCTION
→ WINNER_ANNOUNCE
→ FINAL_REVEAL
→ SETTLEMENT
→ ENDED
```

确定修改内容：

1. 创建对局时先确定本局唯一 `coreWarehouse`，但先不直接进入第 1 轮出价。
2. 服务端下发随机仓候选列表和最终选中仓，前端播放滚动动画。
3. 选中仓展示完成后进入拍卖师情报抽取阶段。
4. 拍卖师情报应用到公共知识层，再进入第 1 轮。
5. 每轮开始时触发公共轮次揭示与角色技能自动触发。
6. 每轮出价结束后进入 `ROUND_FEEDBACK`，保留 5 轮报价轨迹。
7. 未成交则推进下一轮，同仓继续揭示。

### 3. 知识与揭示系统

修改文件范围：

- `packages/match-core/src/match.ts`
- 新增 `packages/match-core/src/knowledge.ts`
- `packages/shared/src/index.ts`

当前 `buildWarehouseSlotViews` 是按索引/进度全局揭示，无法表达“拍卖师公共揭示 + 角色私人揭示”。需要建立知识层。

目标数据结构：

```ts
type ItemKnowledge = {
  knowsExistence: boolean;
  knowsShape: boolean;
  knowsRarity: boolean;
  knowsCategory: boolean;
  knowsValueRange: boolean;
  knowsExactValue: boolean;
  sources: KnowledgeSource[];
};

type KnowledgeSource =
  | "warehouse_intro"
  | "auctioneer_card"
  | "round_reveal"
  | "role_skill"
  | "encyclopedia"
  | "final_reveal";
```

确定修改内容：

1. 对局内维护 `publicKnowledgeByItemId`。
2. 对每名玩家维护 `privateKnowledgeByPlayerId`。
3. 右侧 `WarehouseSlotView` 由“公共知识 + 当前玩家私人知识”合成。
4. 每条情报必须记录来源，并进入中央信息流。
5. 明拍/暗拍模式都使用同一知识系统。

揭示节奏：

| 时机 | 揭示内容 |
| --- | --- |
| 仓库选中后 | 仓名、来源、风险文案、品类倾向 |
| 拍卖师情报 | 随机公共信息，可能揭示具体藏品、统计或最高品质 |
| 每轮开始 | 追加少量公共轮廓/品质/类别/统计 |
| 角色技能 | 按角色向自己或公共层追加信息 |
| 最终开箱 | 全量真值与图标 |

### 4. 角色技能重做

修改文件范围：

- `packages/match-core/src/skills.ts`
- `packages/config/src/data.ts`
- `packages/shared/src/index.ts`
- `apps/web/src/main.tsx`

角色技能从“文本 clue”改成“知识更新 + 文本记录”。

首批角色按我们包装重定：

| 角色 | 当前包装 | 新技能方向 | 触发 |
| --- | --- | --- | --- |
| 鉴定师 | 稳定估值 | 每轮收窄仓库总价值区间，后期更准 | roundStart |
| 走私商 | 高价值单品判断 | 揭示 1 个高价值候选格的轮廓/品质/类别 | roundStart 或 manual |
| 心理师 | 观察对手 | 读取目标本轮可能报价区间或上一轮心理价 | manual |
| 造谣者 | 误导对手 | 暂后置，不进核心 P0；容易偏离“先复刻核心” | P1/P2 |
| 修复师 | 风险识别 | 标记可能有修复费/破损风险的格子 | roundStart |
| 保险商 | 新手稳定 | 改为“避坑账房”：揭示低价值/低品质藏品总价值或数量 | roundStart |

必须先实现 4 个 P0 信息技能：

1. 揭示 N 个未知藏品的轮廓和品质。
2. 揭示 N 个未知藏品的品质。
3. 揭示某类/某品质藏品数量或总价值范围。
4. 揭示一个高价值候选格的轮廓/品质/类别。

技能 UI 要求：

- 自动触发时在中央信息流出现角色头像、技能名、效果描述。
- 对应仓库格播放高亮。
- 被技能影响的格子有来源标记。

### 5. 竞价与成交规则

修改文件范围：

- `packages/match-core/src/auction.ts`
- `packages/match-core/src/bots.ts`
- `packages/shared/src/index.ts`
- `apps/web/src/main.tsx`

保持明拍/暗拍两个模式，暂不恢复次高价、押金明拍、闪拍。

成交规则统一为“领先第二名比例”：

| 轮次 | 程序判断 | UI 解释 |
| --- | --- | --- |
| 第 1 轮 | `leader >= second * 2.0` | 最高价达到第二名 200%，即高出 100% |
| 第 2 轮 | `leader >= second * 1.6` | 最高价达到第二名 160%，即高出 60% |
| 第 3 轮 | `leader >= second * 1.3` | 最高价达到第二名 130%，即高出 30% |
| 第 4 轮 | `leader >= second * 1.1` | 最高价达到第二名 110%，即高出 10% |
| 第 5 轮 | `leader > second` | 最高价高于第二名即成交 |

确定修改内容：

1. `coreCloseThreshold` 与 BidKing 表配置 `[2000,1600,1300,1100,0]` 对齐，UI 文案必须同时说明“达到第二名多少倍”和“高出多少”。
2. 暗拍普通界面不展示具体报价，只展示是否提交、排名、差距级别。
3. 明拍展示当前价、领先者、公开出价记录。
4. 所有人提交后立即结算本轮。
5. 倒计时结束未提交按 0 处理。
6. 第 5 轮同价进入加赛，加赛不新增揭示层，只解决平价。

### 7. UI 流程与布局

修改文件范围：

- `apps/web/src/main.tsx`

目标主流程：

1. 首页/房间选择明拍或暗拍。

   - 右侧：仓库格。
   - 底部：出价、停手、技能/实时情报、表情。
7. 结算时隐藏出价动作栏，注意力集中到原地开箱和收益数字。

随机仓动画要求：

- 候选卡从右向左滚动。
- 有中央选中框。
- 逐步减速。
- 停住后保留 1 秒结果展示。
- 结果以服务端选中仓为准。

修改内容：

- 替换 `scriptedRounds` 拼同仓逻辑。

- 进入第 1 轮前有拍卖师情报。

目标：让情报真正改变仓库格视野。

### 第三轮：局内 UI 与实时情报

目标：让玩家能把信息用于估价。

验收：

- AI 报价能随新信息变化。
- 第 1-4 轮未达成交线会继续揭示同一仓。
- 第 5 轮正确成交或加赛。
- 开箱从亏损红色开始，逐步累计收益。

## 不进入本轮的内容

为了避免再次偏离核心，本轮明确不做：

- 市场交易。
- 商城。
- 活动仓入口。
- 角色养成。
- 付费外观。
- 排位/赛季。
- 好友社交扩展。
- 复杂道具商店。
- 多货币经济。
- 旧设计中的次高价、押金明拍、闪拍主模式。

这些内容不是删除，只是等核心 5 轮玩法稳定后再回收。

- 开始后先进入随机藏宝仓滚动抽选。
- 抽选结果显示仓名、图片、包装文案和风险先验。
- 第 1 轮前出现拍卖师情报。
- 拍卖师情报能改变中央信息流和右侧仓库格。
- 同一个仓库持续 5 轮，不换标的。
- 每轮有新增公共揭示或技能揭示。
- 角色技能不是纯文本，必须改变可见格子或估值信息。
- 左侧显示 4 名玩家、已出价状态和 5 轮排名轨迹。
- 明拍显示金额，暗拍隐藏金额。
- 每轮结束展示排名、差距和成交线是否达成。
- 未成交继续下一轮同仓揭示。
- 成交后右侧仓库格原地逐件开箱。
- 收益从 `-成交价` 开始，亏损红色，转正绿色。
- 实时情报可按品质、类别、轮廓筛选候选藏品。
- AI 的报价随信息增加变化，不表现为固定脚本。

## 第一批具体开发任务

建议下一轮按以下文件顺序开工：

1. `packages/shared/src/index.ts`：补状态、知识、情报卡、实时情报类型。
2. `packages/config/src/schema.ts` / `data.ts`：新增仓型模板、情报牌、角色技能规则。
3. `packages/match-core/src/knowledge.ts`：实现知识合成和揭示应用。
4. `packages/match-core/src/match.ts`：接入随机仓、拍卖师情报、轮次揭示。
5. `packages/match-core/src/skills.ts`：把角色技能改为知识更新。
6. `packages/match-core/src/auction.ts`：统一成交线文案和反馈数据。
7. `packages/match-core/src/bots.ts`：AI 改为按可见知识估价。
8. `apps/server/src/roomManager.ts`：补开局阶段推进和定时事件。
9. `apps/web/src/main.tsx`：补随机仓、情报卡、实时情报和局内布局。
10. `apps/web/src/styles.css`：修正布局密度和移动/桌面适配。

这份计划完成后，再进入实际代码改造。
