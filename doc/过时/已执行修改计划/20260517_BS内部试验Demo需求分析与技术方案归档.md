# B/S内部试验Demo需求分析与技术方案归档

> 项目暂名：珍宝局 / 暗拍大师 / 盲拍奇兵  
> 归档日期：2026-05-17  
> 输入资料：`bitkingdom/doc/多人暗拍博弈游戏详细设计文档.md`、Steam `BidKing` 商店页与玩家评论、`F:/workspace/github/Footnote` 工程与生图流程  
> 目标：先做一版 B/S 内部试验 Demo，但功能目标按“可上线首版”倒推；安全、性能、反外挂规模化指标暂不作为硬门槛。

---

## 1. 结论先行

建议采用传统 Web B/S 架构，而不是沿用 Footnote 的 Phaser 游戏工程作为主框架。

本项目的核心体验是 4 人回合制竞拍、线索阅读、出价控制、结算复盘和局外图鉴，主要由 UI、状态机和实时同步驱动，不依赖复杂物理、地图、角色移动或高频战斗表现。用 React + TypeScript + WebSocket 的传统 Web 栈，开发效率、调试效率、多人状态可视化和后台工具复用都会更高。

Footnote 最值得复用的是“AI 资产生产方法”，不是主游戏引擎：

- 可复用：资产分类、提示词模板、锚定图、批量生成脚本思路、生成后 QC 清单。
- 不建议直接复用：Phaser 场景系统、叙事冒险状态结构、像素赛博 UI 风格。
- 需要改造：从“像素赛博冒险”改为“三国珍宝 + 动物拟人拍卖场”的轻竞技桌游包装。

推荐技术基线：

```text
客户端：React + TypeScript + Vite + CSS Modules/Tailwind（二选一）
服务端：Node.js + TypeScript + Fastify + Socket.IO
数据层：PostgreSQL + Prisma
缓存/队列：Demo 阶段暂不引入 Redis，后续按并发扩展
共享包：shared types + config schema + match reducer
测试：Vitest + Playwright
部署：Docker Compose 本地/内网环境
```

首版 Demo 必须坚持服务端权威结算。即使暂不做高强度反作弊，也不能把货柜真实价值、完整藏品、私人线索和随机种子直接下发给客户端，否则会立即复现竞品评论里“透明仓”“外挂像知道答案”的信任问题。

## 2. 竞品观察与机会点

### 2.1 Steam 页面基线

Steam 商店页显示，`BidKing` 是一款多人竞拍与收藏鉴定策略游戏，核心卖点包括 4 人心理战、暗标竞拍、角色信息技能、逐步开箱揭晓和收藏柜展示。商店页发布日期为 2026-04-15，页面同时展示近期评价和总体评价均偏低：截至本次调研，近期评价为 Mixed，全量评价为 Mostly Negative。  

SteamDB 页面显示该游戏使用 Unity Engine，支持 Windows，标签集中在 Strategy、Casual、Simulation、Massively Multiplayer、Trading、Resource Management 等方向。

### 2.2 玩家负反馈聚类

基于 Steam 商店页评价摘要与社区评论，本项目需要重点规避以下问题：

| 负反馈类别 | 玩家感知 | 对本项目的要求 |
|---|---|---|
| 服务器差、卡顿、掉线 | 局内出价不可信，货币/收益可能丢失 | Demo 也要做服务端权威、事件流水、可重放结算 |
| 外挂/信息透明 | 对手像提前知道箱内价值 | 真实价值、完整藏品、随机种子禁止下发客户端 |
| 假匹配/AI 对手 | 玩家觉得自己在和全知 AI 玩 | 机器人行为必须有限信息、可解释、可调笨 |
| 掉率与经济争议 | 稀有物过低、付费/通行证被骂 | 首版不接真实付费，局内公平和图鉴掉落分离 |
| AI 素材廉价感 | 被评价为低质量 AI 资产 | 建立锚定图、统一风格、生成后人工筛选与修图 |
| 窗口/分辨率适配差 | UI 缩放后无法操作 | Web 首版必须做桌面与移动响应式验收 |
| 封号/吞币争议 | 运营可信度崩坏 | 即使不做完整安全，也保留交易流水与客服查询码 |

机会点很明确：玩法原型有吸引力，但信任、解释、经济和 UI 适配决定留存。我们的设计文档已经把“输得明白”“公平竞技”“复盘解释”作为核心目标，这是正确方向。

## 3. Demo 产品目标

### 3.1 Demo 定位

这不是一次低保真原型，而是一版“可上线首版的功能型 Demo”：

- 可供内部多人真实联机试玩；
- 可持续跑 5 轮完整对局；
- 可验证暗拍、明拍、次高价拍卖的乐趣；
- 可验证线索、技能、误导、机器人是否形成心理博弈；
- 可展示基础美术包装、开箱反馈、图鉴点亮和结算复盘；
- 暂不承诺商业级安全、压测、复杂运营后台和高并发指标。

### 3.2 核心体验目标

| 目标 | 验收观察 |
|---|---|
| 玩家能理解为何出价 | 每轮至少有公共线索、私人线索和风险提示 |
| 玩家能产生心理博弈 | 有暗标、表情/状态、角色技能和误导信息 |
| 玩家能接受亏损 | 单轮结算解释成交价、真实价值、修复费、赝品原因 |
| 玩家愿意再来一局 | 5 轮节奏不拖沓，开箱有反转，最终排名有悬念 |
| 玩家相信公平 | 复盘能解释信息来源，机器人不表现为全知 |
| 内容可扩展 | 货柜、藏品、线索、角色、技能都走配置 |

### 3.3 首版非目标

| 暂缓内容 | 原因 |
|---|---|
| 真实充值、战令、交易市场 | 内部 Demo 不需要，且会放大信任风险 |
| 大规模反作弊系统 | 先用服务端权威和隐藏信息边界解决根问题 |
| 高并发匹配池 | 内部 Demo 可用房间码 + 简单快速匹配 |
| 3D 收藏柜 | 首版用 2D 图鉴和藏品卡展示即可 |
| 语音聊天 | 审核和社交风险高，先用表情/快捷语 |
| 复杂运营后台 | 先做配置热加载/管理脚本，后台后置 |

## 4. 功能需求拆分

### 4.1 账号与入口

Demo 阶段支持轻账号：

- 访客登录：输入昵称即可进入；
- 本地保存最近昵称；
- 可选开发者登录：方便测试指定用户、指定角色、指定房间；
- 后续预留微信/Steam/OpenID 适配层，但本期不接平台 SDK。

主界面首版入口：

- 快速匹配；
- 创建/加入房间；
- 角色选择；
- 图鉴；
- 任务；
- 设置；
- 调试入口，仅开发环境显示。

### 4.2 房间与匹配

必须支持两种开局方式：

| 方式 | 用途 | 规则 |
|---|---|---|
| 快速匹配 | 模拟正式体验 | 4 人满员开局，不足时机器人补位 |
| 房间码 | 内部试玩 | 房主创建房间，其他人输入房间码加入 |

房间基础规则：

- 4 个座位；
- 玩家可选择角色；
- 房主可选择拍卖模式池、轮数、机器人数量；
- 准备状态齐全后开始；
- 掉线玩家 30 秒内保留座位，超时由机器人托管。

### 4.3 对局主循环

首版固定 4 人 5 轮：

```text
入场
→ 展示本轮货柜
→ 分发公共线索/私人线索
→ 情报与技能阶段
→ 竞价阶段
→ 成交
→ 开箱逐步揭晓
→ 单轮结算与复盘
→ 下一轮
→ 最终排名与奖励
```

每轮建议目标时长：

| 阶段 | 时长 | 说明 |
|---|---:|---|
| 货柜展示 | 8 秒 | 让玩家看来源、估值、风险 |
| 情报阶段 | 15 秒 | 技能、线索、误导 |
| 竞价阶段 | 25-35 秒 | 根据拍卖模式变化 |
| 成交展示 | 3 秒 | 展示赢家和价格 |
| 开箱 | 8-12 秒 | 逐步揭晓，可跳过但默认同步节奏 |
| 结算 | 8 秒 | 利润、线索命中、排名变化 |

### 4.4 拍卖模式

首版实现 3 个模式：

| 模式 | 优先级 | 核心要求 |
|---|---:|---|
| 明拍 | P0 | 所有人可见当前价，支持加价、停手、倒计时延长 |
| 暗标 | P0 | 同时提交一次最终报价，其他人只看到“已出价/犹豫”等状态 |
| 次高价拍卖 | P0 | 最高出价者成交，支付第二高价 + 固定加价单位 |

暂缓模式：

- 押金拍卖；
- 闪拍；
- 自定义娱乐规则。

### 4.5 货柜、藏品与线索

配置规模建议：

| 内容 | Demo 数量 | 目标 |
|---|---:|---|
| 货柜模板 | 30 个 | 覆盖稳定、中风险、高风险、套装、特殊事件 |
| 藏品 | 100 个 | 破烂、普通、精良、稀有、传说、赝品 |
| 套装 | 5 套 | 五虎兵器、宫廷玉器、古战场遗物等 |
| 公共线索模板 | 60 条 | 来源、品类、风险、估值区间 |
| 私人线索模板 | 80 条 | 价值、赝品、破损、套装、对手状态 |

关键规则：

- 每轮至少 1 条公共线索；
- 每名玩家至少 1 条私人线索；
- 线索必须能解释结果，避免纯随机暴雷；
- 高价值陷阱必须至少存在可识别风险；
- 线索准确率、误差范围和受技能影响情况要可配置。

### 4.6 角色与技能

首版 6 个角色：

| 角色 | 定位 | 首版技能实现 |
|---|---|---|
| 鉴定师 | 稳定估值 | 查看总价值区间，误差较小 |
| 走私商 | 单品嗅觉 | 查看最高价值单品品质区间 |
| 心理师 | 观察对手 | 查看一名玩家出价意愿区间 |
| 造谣者 | 信息误导 | 投放一条模糊假情报 |
| 修复师 | 破损风险 | 预估是否存在高修复费物品 |
| 保险商 | 防暴亏 | 本轮亏损超过阈值后返还部分损失 |

设计底线：

- 技能不能直接显示完整答案；
- 技能不能带来局内本金优势；
- 技能必须走服务端结算；
- 技能产生的信息要进入局后复盘。

### 4.7 机器人

机器人是 Demo 必需品，不是填充物。

机器人目标：

- 补足对局；
- 带动竞价节奏；
- 让新手看到“有人停手、有人上头、有人诈唬”；
- 内部压测不同策略。

机器人类型：

| 类型 | 行为 |
|---|---|
| 保守型 | 根据估值下限出价，容易停手 |
| 激进型 | 愿意追高，偶尔过价 |
| 线索型 | 根据分配到的线索合理估价 |
| 误导型 | 会使用表情和假情报 |
| 新手陪练型 | 不连续精准压制玩家 |

关键要求：机器人只能使用自己被分配到的信息，不得读取完整货柜真实价值来“演”。服务器内部可以生成真实货柜，但机器人策略输入必须是过滤后的 `BotKnowledge`。

### 4.8 开箱与结算复盘

开箱必须做出节奏：

```text
低价值杂物
→ 中价值藏品
→ 风险/赝品/破损
→ 稀有或套装
→ 总估值与利润
```

单轮结算展示：

- 成交价；
- 真实价值；
- 修复费/赝品损失；
- 最终利润；
- 本轮排名变化；
- 线索命中列表；
- 本轮称号，例如“极限捡漏”“成功诱导”“高价接盘”。

最终结算展示：

- 4 人排名；
- 净资产曲线；
- 最佳操作；
- 最大失误；
- 新增图鉴；
- 经验/任务进度。

### 4.9 局外成长

Demo 阶段保留轻量局外系统：

- 玩家等级；
- 图鉴点亮；
- 每日/成就任务的假数据或本地进度；
- 角色解锁状态；
- 外观系统只做入口和数据结构，不接真实购买。

## 5. 技术架构方案

### 5.1 总体架构

```text
Browser Client
  ├─ React UI
  ├─ WebSocket match channel
  ├─ HTTP API client
  └─ Local cache

Node Server
  ├─ HTTP API: account, lobby, config, profile
  ├─ Socket Gateway: room, match, bid, sync
  ├─ Match Runtime: authoritative state machine
  ├─ Bot Engine
  ├─ Config Loader
  └─ Persistence Adapter

Database
  ├─ player/profile
  ├─ match summary
  ├─ event log
  ├─ transaction log
  └─ progression
```

### 5.2 客户端选型

推荐：

```text
React + TypeScript + Vite
Zustand：客户端局部状态
TanStack Query：HTTP 数据缓存，可选
Socket.IO Client：实时连接
CSS Modules 或 Tailwind：二选一
Vitest + Testing Library：组件/逻辑测试
Playwright：端到端联机流程测试
```

不推荐首版使用 Phaser 作为主框架：

- 局内大部分是表单、列表、卡片、弹窗、状态条和动画；
- DOM/CSS 更适合响应式、可访问性、文本排版和后台工具复用；
- 多人状态调试时，React DevTools + 网络面板比游戏引擎场景更直观；
- 后续如果需要强开箱特效，可单独用 Canvas/WebGL 小组件，而不是全项目 Phaser 化。

### 5.3 服务端选型

推荐：

```text
Node.js 20+
TypeScript
Fastify
Socket.IO
Prisma
PostgreSQL
Zod 或 Valibot 做 runtime schema 校验
```

选择理由：

- 与客户端共享 TypeScript 类型；
- Socket.IO 对房间、断线重连、内网调试友好；
- Fastify 比较轻，适合先快速搭 Demo；
- Prisma 能快速建表、迁移、生成类型；
- PostgreSQL 让事件流水、复盘、后续后台查询更可靠。

### 5.4 Monorepo 目录建议

建议在 `bitkingdom` 下新建工程结构：

```text
bitkingdom/
  apps/
    web/                  # React 客户端
    server/               # Fastify + Socket.IO 服务端
  packages/
    shared/               # 共享类型、协议、工具函数
    config/               # 货柜/藏品/线索/角色配置与 schema
    match-core/           # 对局状态机、结算、拍卖规则
  assets/
    source/               # 生图源文件、锚定图、提示词
    generated/            # AI 生成原图
    approved/             # 人工筛选后入包资源
  tools/
    assetgen/             # 生图脚本，参考 Footnote text2pic
    config-check/         # 配置校验脚本
  doc/
    多人暗拍博弈游戏详细设计文档.md
    20260517_BS内部试验Demo需求分析与技术方案归档.md
```

### 5.5 对局状态机

服务端维护权威 `MatchState`，客户端只发意图：

```text
Client Intent:
  joinRoom
  ready
  selectRole
  useSkill
  submitBid
  passAuction
  sendEmote
  requestSnapshot

Server Event:
  roomUpdated
  matchStarted
  phaseChanged
  clueReceived
  bidStateUpdated
  auctionResolved
  itemRevealed
  roundSettled
  matchEnded
```

核心原则：

- 客户端不能直接修改状态；
- 服务端校验阶段、剩余资金、技能冷却、出价合法性；
- 所有影响结算的操作写入事件日志；
- 每个阶段有明确超时处理；
- 机器人作为服务端内部玩家提交同样的 intent。

### 5.6 隐藏信息边界

为避免“透明仓”：

| 数据 | 客户端可见性 |
|---|---|
| 货柜来源、公开估值、公开标签 | 所有人可见 |
| 自己的私人线索 | 仅本人可见 |
| 其他玩家私人线索 | 不下发 |
| 完整藏品列表 | 开箱前不下发 |
| 真实价值 | 结算前不下发 |
| 随机种子 | 对局结束后可用于复盘，不在局中下发 |
| 机器人完整判断依据 | 不下发，复盘可摘要展示 |

### 5.7 数据模型草案

```typescript
interface Match {
  id: string;
  status: 'lobby' | 'playing' | 'settled';
  roundIndex: number;
  players: MatchPlayer[];
  configId: string;
  createdAt: string;
}

interface Round {
  id: string;
  matchId: string;
  index: number;
  auctionMode: 'open' | 'sealed' | 'second_price';
  containerInstanceId: string;
  phase: RoundPhase;
  startsAt: string;
  endsAt: string;
}

interface ContainerInstance {
  id: string;
  templateId: string;
  publicInfo: PublicContainerInfo;
  hiddenItems: ItemInstance[];
  publicClues: Clue[];
  privateCluesByPlayerId: Record<string, Clue[]>;
}

interface BidIntent {
  matchId: string;
  roundId: string;
  playerId: string;
  amount: number;
  clientSeq: number;
}

interface TransactionLog {
  id: string;
  matchId: string;
  roundId?: string;
  playerId: string;
  reason: string;
  amountBefore: number;
  amountChange: number;
  amountAfter: number;
  createdAt: string;
}
```

### 5.8 配置格式

配置用 YAML 或 JSON 均可。考虑策划可读性，推荐 YAML；运行时构建为 JSON。

配置类型：

- `containers.yaml`：货柜模板；
- `items.yaml`：藏品；
- `clues.yaml`：线索模板；
- `roles.yaml`：角色与技能；
- `auctionModes.yaml`：拍卖模式参数；
- `botProfiles.yaml`：机器人策略参数；
- `progression.yaml`：等级、任务、图鉴奖励。

每次启动或构建前运行配置校验：

```text
npm run validate:config
```

校验内容：

- 引用 ID 存在；
- 价值区间合法；
- 稀有度权重合计合理；
- 每个货柜至少有可解释线索；
- 每个角色技能有冷却和复盘文案；
- 新手货柜不包含过高赝品惩罚。

## 6. 美术与生图流程

### 6.1 包装方向

沿用详细设计文档推荐方向：

```text
三国珍宝 + 动物拟人拍卖场
```

视觉关键词：

- 乱世商会；
- 黑市拍卖；
- 三国珍宝；
- 动物拟人掌柜；
- 木、漆、铜、玉、竹简、锦缎；
- 轻松互坑，不做厚重战争。

首版应避免直接模仿 BidKing 的现实货柜/集装箱包装，尽量形成自有识别。

### 6.2 Footnote 流程可复用点

Footnote 的 `workflows/reusable/text2pic` 提供了可参考的 API 封装与批量生成结构：

- `doc.md`：记录 Gemini 2.5 Flash Image 的文生图/图生图接口；
- `image_gen.py`：封装文生图和图生图；
- `batch_generate.py`：按背景、角色、物件、特效、UI 分类批量生成；
- `design/production/art/提示词模板.md`：强调锚定图、生成/编辑规则、QC 清单；
- `design/production/art/UI设计规范.md`：提供 UI 单位、尺寸、状态规范。

本项目改造原则：

- API Key 必须使用环境变量，不要硬编码进脚本；
- 提示词前缀改为本项目风格；
- 资产先进入 `generated`，人工筛选后进入 `approved`；
- 角色表情、藏品破损、开箱状态变体优先使用 Edit 模式；
- UI 图标尽量用代码/图标库实现，只有复杂纹理面板和藏品图用生图。

## 7. 阶段计划

### 7.1 M0：工程骨架与配置验证

目标：确定技术路线并跑通最小闭环。

交付：

- Monorepo 初始化；
- Web/Server 启动；
- WebSocket 连接；
- 共享类型包；
- 配置 schema；
- 1 个本地模拟货柜；
- 1 个开发调试页面。

验收：

- 浏览器能连接服务器；
- 服务端能创建房间；
- 客户端能展示权威状态快照；
- 配置校验可执行。

### 7.2 M1：单机/本地 4 人核心循环

目标：验证玩法循环。

交付：

- 5 轮对局状态机；
- 明拍、暗标、次高价拍卖；
- 货柜生成；
- 线索分发；
- 开箱揭晓；
- 单轮结算；
- 3 类机器人。

验收：

- 一个真实玩家 + 3 个机器人可完成整局；
- 每轮有赚亏；
- 复盘能解释至少 80% 的亏损原因；
- 内部试玩能连续玩 3 局。

### 7.3 M2：真实多人联机 Demo

目标：完成 B/S 试验版核心。

交付：

- 快速匹配；
- 房间码；
- 4 人实时状态同步；
- 出价合法性校验；
- 断线重连基础版；
- 事件日志；
- 交易流水；
- 最终结算。

验收：

- 4 个浏览器同时完成 5 轮；
- 刷新后能恢复当前局面；
- 玩家无法在网络包中看到未揭晓完整藏品；
- 结算结果可由事件日志重放。

### 7.4 M3：上线级体验补全

目标：让 Demo 看起来像首版产品。

交付：

- 6 个角色与技能；
- 图鉴；
- 任务；
- 新手引导；
- 风险提示；
- 美术首版；
- 音效占位；
- Playwright 冒烟测试。

验收：

- 新用户 1 局内能理解“看线索、出价、停手、复盘”；
- 局内主要按钮响应式适配桌面与移动；
- 无明显 UI 重叠和文字溢出；
- 角色技能在复盘中可解释。

## 8. 验收清单

### 8.1 核心玩法

- [ ] 4 人局可完整跑完 5 轮；
- [ ] 明拍、暗标、次高价拍卖都可独立完成；
- [ ] 每名玩家每轮至少获得 1 条私人线索；
- [ ] 每轮至少有 1 条可用于判断的公共线索；
- [ ] 出价不能超过剩余资金；
- [ ] 次高价拍卖支付价格正确；
- [ ] 开箱逐步揭晓，不提前暴露真实价值；
- [ ] 单轮和最终排名计算正确。

### 8.2 公平与可信

- [ ] 客户端看不到未揭晓藏品完整列表；
- [ ] 客户端看不到其他玩家私人线索；
- [ ] 机器人只基于过滤后的信息行动；
- [ ] 每次资金变化有交易流水；
- [ ] 每局有事件日志；
- [ ] 结算可重放；
- [ ] 断线重连能恢复当前快照。

### 8.3 UI 体验

- [ ] 桌面 1366×768 可完整操作；
- [ ] 移动竖屏 390×844 可完整操作；
- [ ] 所有主要按钮触控区域不小于 44×44；
- [ ] 长文本线索不溢出；
- [ ] 竞价阶段倒计时和当前行动清晰；
- [ ] 结算页能看懂利润来源；
- [ ] 图鉴新增有反馈。

### 8.4 内容生产

- [ ] 资产使用统一提示词前缀；
- [ ] 生成资产无文字、无水印；
- [ ] 角色、藏品、货柜风格一致；
- [ ] 生成图进入 `generated`，审核后进入 `approved`；
- [ ] 配置引用的资源文件存在；
- [ ] 低质量 AI 图不进入正式包。

## 9. 主要风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 玩法被认为“全靠运气” | 留存低 | 每轮保证有效线索，局后解释线索命中 |
| 玩家觉得机器人作弊 | 信任崩 | 机器人只用过滤信息，并在调试页可查看决策输入 |
| 暗标节奏太闷 | 参与感低 | 显示已出价、犹豫、表情、技能提示，不显示具体金额 |
| 明拍上头亏损太狠 | 挫败强 | 新手推荐停手价、保险商、单轮暴亏软限制 |
| AI 美术廉价 | 首屏劝退 | 建立锚定图与人工 QC，不用未筛选生成图 |
| 配置快速膨胀 | 维护困难 | schema 校验、ID 规范、配置预览页 |
| WebSocket 同步复杂 | Bug 多 | 服务端 reducer + 事件日志 + 客户端快照校准 |

## 10. 后续开发建议

第一步不要急着做完整局外系统。建议先开 `M0 + M1`，把核心对局状态机、机器人、三种拍卖和复盘解释打穿。只要这一层成立，后续美术、图鉴、任务、房间和联机都是加厚；如果这一层不好玩，外层做再漂亮也救不了。
