# 后台管理回看需求分析与实现记录

> 日期：2026-05-17  
> 目标：增加内部后台管理能力，可以回看每场对局的结果和过程。  
> 范围：先做 B/S Demo 内部管理页，不做正式账号权限、数据库和运营后台。

## 1. 需求分析

### 1.1 使用对象

- 策划：回看每轮货柜、模式、赢家、收益和玩家行为，判断节奏是否符合设计。
- 程序/测试：核对服务端事件、货币流水、结算结果是否一致。
- 内部试玩组织者：快速查看每场谁赢、哪里亏、玩家是否理解规则。

### 1.2 必要能力

| 能力 | 说明 | 本轮实现 |
|---|---|---|
| 对局列表 | 展示服务器当前保存的所有已开始对局 | 已实现 |
| 对局概览 | 房间码、状态、轮次、玩家、赢家、事件数、流水数 | 已实现 |
| 结果回看 | 玩家最终净资产排序、赢家标记 | 已实现 |
| 过程回看 | 按轮查看拍卖模式、货柜、赢家、成交、真值、收益 | 已实现 |
| 决策依据回看 | 按轮查看公共线索、每名玩家的私有/技能线索、完整出价快照 | 已实现，本轮复核补强 |
| 行为日志 | 按轮查看阶段切换、出价、停手、技能、开箱、结算 | 已实现 |
| 货币流水 | 按轮查看押金、成交付款、修复费、保险返还 | 已实现 |
| 自动刷新 | 对进行中的对局可持续刷新 | 已实现，5秒轮询 |

### 1.3 暂缓能力

- 管理员登录、权限和审计。
- 数据库持久化与跨进程查询。
- 对局搜索、筛选、导出。
- 可拖动进度条式回放。
- 后台主动干预对局，例如强制结算、踢人、补偿。

## 2. 数据来源分析

现有服务端已经具备回看基础：

- `MatchRuntimeState.events`：记录阶段、出价、技能、开箱、结算等事件。
- `MatchRuntimeState.transactions`：记录所有货币变化。
- `MatchRuntimeState.roundHistory`：记录每轮结果、货柜、赢家、真值和收益。
- `FinalMatchSummary`：记录最终排名、资产曲线、奖励和复盘摘要。

因此本轮不新增复杂存储，只在服务器运行期间从内存房间表中聚合管理端数据。

## 3. 本轮实现

- 新增共享类型：`AdminMatchListItem`、`AdminMatchDetail`、`AdminRoundReplay`。
- 新增服务端接口：
  - `GET /api/admin/matches`
  - `GET /api/admin/matches/:matchId`
- 新增前端后台入口：顶部按钮“后台管理”，也支持 `/admin` 路径进入。
- 新增后台页面：
  - 左侧对局列表；
  - 右侧对局概览；
  - 玩家净资产排序；
  - 逐轮过程卡片；
  - 每轮公共线索、私有/技能线索、出价快照、参与者结算拆分；
  - 事件时间线；
  - 货币流水。

## 4. 复核补强

提交后复核发现，原后台可以看到轮次结果、事件和流水，但结束后的历史轮次没有独立归档“当轮玩家看到的线索”和“完整出价快照”。如果只依赖当前公开状态，结束后很难解释玩家为什么高价或低价出手。

本轮补强：

- `RoundHistoryEntry` 增加公共线索、每名玩家私有/技能线索、完整出价记录和结算对象。
- 管理端 `AdminRoundReplay` 直接聚合这些历史快照，进行中的轮次也会暴露当前快照。
- 后台逐轮卡片新增线索、出价和参与者收益拆分展示。
- 增加核心测试覆盖，保证回合归档包含线索、出价和结算。

## 5. 后续建议

- 引入轻量管理员口令或内网白名单。
- 将结束对局写入 SQLite/Postgres，避免服务器重启后丢失。
- 增加筛选：日期、房间码、玩家名、拍卖模式、亏损大于阈值。
- 增加导出：JSON/CSV，方便数值分析。
- 增加“关键事件高亮”：超过安全价、押金亏损、保险触发、假情报影响。
