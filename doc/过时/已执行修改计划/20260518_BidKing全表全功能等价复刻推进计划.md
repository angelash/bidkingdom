## 4. 服务端需求分析与设计

当前 demo 的服务端主要是 `roomManager`：创建房间、加入/重连、出价、放弃、用技能、机器人推进、推送快照。全功能复刻必须把服务端扩展为局外成长和经济系统的权威状态源，否则商店、任务、邮件、门票、排行和社交即使有 UI，也无法做到原流程等价。

### 4.1 服务端必须承担的状态

- 档案：`playerId/sessionId`、昵称、头像、等级、经验、货币、票券、设置、图鉴、背包、皮肤。
- 条件：统一解释 `Condition / Access / Mission / Achievement / ShopItem.front`。
- 奖励：统一解释奖励数组，发放经验、货币、道具、票券、头像、皮肤、邮件附件、通行证经验。
- 经济账本：记录商品购买、比赛奖励、邮件领取、任务领奖、票券恢复/消耗，支持幂等 `sourceId`。
- 门票：服务端恢复、封顶、购买刷新，并在进入竞拍前扣减。
- 商店：维护 `Shop / ShopItem / ItemRestock / ExchangeRestock` 的商品池、限购、刷新和价格扣减。
- 任务成就：消费比赛、技能、图鉴、购买、交易、邮件等事件，按条件结算进度和奖励。
- 邮件公告：维护 inbox、已读、附件领取、公告有效期、错误码映射。
- 排行交易：维护排行榜快照、领奖记录、交易所/拍卖行本地模拟订单。
- 社交协会：维护好友、协会、职位权限、地区积分、表情、昵称和敏感词过滤。
- 战斗集成：校验战前道具、角色、皮肤、地图入口；比赛结束后统一写事件并结算成长。

### 4.2 服务端模块设计

建议把 `roomManager` 降级为实时战斗房间模块，在 `apps/server/src/services/` 下拆出业务服务：

```text
profileService.ts         # 档案、等级、设置、图鉴、背包汇总
configBootstrapService.ts # 表版本、入口开关、公告、错误码、资源键
conditionService.ts       # Condition / Access / Mission 条件解释
rewardService.ts          # 奖励数组解释与发放计划
economyService.ts         # 货币、票券、库存 transaction ledger
ticketService.ts          # Ticket 恢复、消耗、购买刷新
missionService.ts         # Mission / Achievement 事件结算
shopService.ts            # Shop / ShopItem / Restock 购买刷新
mailService.ts            # Mail 模板实例、附件领取
rankService.ts            # Rank / RankReward 快照和领奖
tradeService.ts           # Exchange / AuctionHouse 本地订单模拟
socialService.ts          # Friend / Guild / Area 本地模拟
textGuardService.ts       # DirtyWords、昵称/聊天过滤
```

存储先抽象成 `Store` 接口，demo 默认使用 `memoryStore` 和 `jsonFileStore`，后续再替换 SQLite 或正式数据库。所有发奖、扣费和库存变更必须走 `economyService.applyTransaction(sourceId, changes)`，避免重连、刷新、重复结算造成多发。

### 4.3 接口与事件草案

- `GET /api/bootstrap`：配置版本、入口开关、公告、错误码、资源键。
- `GET /api/profile`：玩家档案、货币、票券、任务、邮件未读数、商店摘要。
- `POST /api/profile/settings`：保存语言、音量、画质、输入等设置。
- `POST /api/ticket/refresh`：刷新并返回最新票券状态。
- `POST /api/shop/buy`：购买商品，校验 `Access / front / price / buycounts`。
- `POST /api/shop/refresh`：消耗刷新资源并重建商品池。
- `POST /api/mail/claim`：领取附件，写入账本。
- `POST /api/mission/claim`：领取任务/成就奖励。
- `POST /api/battle/items/equip`：战前携带道具，校验库存、上限和入口条件。
- `POST /api/trade/order`：创建本地模拟交易/拍卖行订单。
- `POST /api/rank/claim`：领取排行奖励。
- socket 推送：`profileUpdated / ticketUpdated / shopUpdated / mailUpdated / missionUpdated / rankUpdated`。

`createRoom/startMatch` 的新流程：

1. 校验 `Access`：地图、角色、皮肤、道具是否可用。
2. 刷新并校验 `Ticket`。
3. 消耗门票，写入 transaction。
4. 固化本局 loadout：角色、皮肤、战前道具、地图、随机种子。
5. 比赛结束后写 match result event，由任务、排行、图鉴、奖励服务统一结算。
6. 推送最新 profile snapshot 给前端。

### 4.4 服务端验收标准

- 关闭浏览器再进入，货币、票券、任务、邮件、商店限购和图鉴状态不丢。
- 竞拍入口必须由服务端扣票，票不足不能开局。
- 同一场比赛结算重复推送不会重复发奖励。
- 商品购买、邮件领取、任务领奖都有 transaction 记录。
- 前端所有经济数字来自 profile snapshot，不再由本地手写状态自行计算。
- 管理后台能查看玩家档案、账本、任务事件、邮件状态、商店状态和本局战斗结果。

## 5. 执行顺序

1. 建服务端底座：`Store + profileService + economyService + ticketService`，让 profile、票券和账本从服务端出。
2. 写 `conditionEngine` 和 `rewardEngine`，让任务、商店、成就、入口解锁共享同一套解释器。
3. 改 `createRoom/startMatch`：接 `Ticket` 消耗、`Access` 校验、战前 loadout 固化和比赛结束事件。
4. 接 `Mission / Achievement / LevelUp / Mail`：比赛事件驱动任务、成就、等级和邮件附件领取。
5. 接 `Shop / ShopItem / ItemRestock / Exchange_Restock / Ticket`：补购买、限购、刷新、价格扣减和奖励发放。
6. 接 `Rank / RankReward / Trade / AuctionHouse`：补排行榜、领奖、交易所和拍卖行本地模拟账本。

8. 最后接 `Language / LanguageListen / DirtyWords / Sim / Sound`：完善本地化 key、语音注册、输入过滤、模拟和音频触发。
9. 拆前端 UI 结构：按 `Battle* / BattlePrev* / Shop* / Mail* / Task* / Rank*` 等原类群分模块复刻，并全部读取服务端 snapshot。

## 6. 验收标准

- 52 张表都有对应安全运行表或明确的“只作本地参考、不进入提交”说明。
- 每个已显示入口都能追到一张或多张配置表。
- 所有数值曲线、行数、关联 ID 通过 parity 校验。
- 所有 UI 文案、角色名、物品名、资源名使用《珍宝局》包装。
- 所有经济状态、票券、任务、邮件、商店限购和领奖状态由服务端 profile snapshot 驱动。
- 所有发奖、扣费、购买、领奖、比赛结算都有 transaction 或 event 记录，重复请求不重复发放。
- 不提交 `reverse/`、原始商业文本、原始资源、反编译源码。
