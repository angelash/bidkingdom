# 功能逻辑评估与本轮补强记录

> 日期：2026-05-17  
> 范围：只评估功能性逻辑，不评估美术资源数量与质量。  
> 对照目标：`多人暗拍博弈游戏详细设计文档.md`、`20260517_BS内部试验Demo需求分析与技术方案归档.md`、`20260517_详细设计逐模块实现核对与迭代计划.md`

## 1. 当前已基本达标

- 4人房间、机器人补位、房间码加入、断线后本地会话重连已可用。
- 服务端权威对局、隐藏信息快照、事件日志与货币流水接口已建立。
- 默认Demo已经按文档样例5轮脚本出货柜：明拍、暗标、次高价、押金明拍、闪拍。
- 公共线索、私人线索、技能线索、假情报、线索复盘已经接入。
- 6个角色、主动技能、冷却、每局次数已经接入。
- 局外轻量反馈已有本地图鉴、任务、等级、段位、收藏柜和最终奖励。
- 局内UI已有拍卖规则、当前领先者、出价记录、推荐安全价、风险提示、开箱、单轮结算和最终复盘。

## 2. 本轮发现的主要功能缺口

| 优先级 | 问题 | 影响 |
|---|---|---|
| P0 | Web类型检查失败：`artAssets.ts` 的生成藏品图标索引存在 `undefined` 类型风险 | 构建链路不稳定，后续改动无法可靠验收 |
| P0 | 修复费只进入单轮利润文案，没有实际扣现金 | 最终资产排名可能和结算解释不一致，破坏“可信结算” |
| P0 | 押金明拍首次出价没有把押金预留进可出价资金 | 玩家可能先付押金再按原现金满额成交，造成不合理负现金 |
| P1 | 押金明拍只展示赢家押金，未解释非赢家押金损耗 | 玩家看不懂自己为什么少了钱 |
| P1 | 暗标/次高价/闪拍的停手记录可重复写入 | 出价记录和回放流水会变脏 |
| P1 | 技能目标由客户端自动挑选，玩家无法明确选择读心/观察对象 | 角色技能的心理博弈表达不足 |
| P2 | 技能次数耗尽后按钮仍可能可点，只能依赖服务端报错 | 操作反馈不够干净 |

## 3. 本轮已开发补强

- 核心结算新增每名玩家的分账结果：成交价、押金支付、押金返还、修复费、保险返还、真值、套装加成、净变化和结果标题。
- 修复费改为服务端实际货币流水 `repair_cost_paid`，最终净资产与单轮利润保持一致。
- 押金明拍出价上限改为“当前现金 - 待支付押金”，避免押金后仍按原现金满额出价。
- 非赢家押金返还进入分账复盘，能展示“试探成本”。
- 暗标类停手记录改为幂等更新，闪拍已提交后不能撤回。
- 前端单轮结算增加“本轮分账”表，按玩家展示每笔关键成本和净变化。
- 前端增加技能目标下拉选择，技能次数为0时直接禁用按钮。
- 修复生成藏品图标映射的类型错误。
- 新增核心测试覆盖押金明拍资金上限、修复费扣款、每人押金分账和暗标停手幂等。

## 4. 仍建议后续补的非资源功能

- 局外图鉴、任务、等级、段位从本地存储改为账号级持久化。
- 最终复盘增加面向当前玩家的逐轮原因总结，例如“哪轮被假情报误导”“哪轮出价超过安全价”。
- 回放面板从原始事件列表升级为可读时间线，直接串起技能、出价、押金、结算。
- 新手第一局提示改为阶段脚本，可随样例5轮逐步解锁，不只是可关闭的通用提示。
- 数据埋点补充技能使用率、停手点、超过安全价次数、拍卖模式胜亏分布。
- 断线重连补服务端身份令牌或账号绑定，避免服务器重启后只能依赖本地房间码。
