## 一、BidKing 原实现技术结构

### 1. 启动与热更

本地 Unity 构建是 Mono/热更组合：

- 入口很薄：`Assembly-CSharp` 只有少量启动代码。
- 主要业务逻辑在 `StreamingAssets/dll/Scripts.dll.bytes`。
- `NotHotUpdate` 负责启动、热更程序集加载、Base64 配置读取、XOR 解密。
- DLL 还原逻辑证据：
  - `FirstTools.LoadAssembly()`
  - `CryptoUtils.DecryptRaw(..., "ryrs")`
  - `Base64ToTxt.DecodeBase64ToTxt()`

对复刻的意义：

- 游戏核心并不依赖 Unity 场景硬编码，主要由表驱动和热更 C# 驱动。
- 我们前端/服务端分离实现可继续保留，只要状态机、配置和 UI 表达对齐即可。

### 2. UI 与对局主模块

关键反编译类：

- `Battle_Main.cs`
  - 注册 `E_OnRoundStart`、`E_OnRoundStartBid`、`E_OnPlayerBid`、`E_OnPrevRoundPlayerAuction`、`E_OnBattleAuctionCallback` 等事件。
  - 管理出价按钮、弃权、倒计时、玩家轮次记录、技能描述、最终开箱、总价与利润。
- `Auction.cs`
  - 管理仓库格与藏品格对象池。
  - `InitGrids()` 创建格子，`RefreshItem()` 把 `GridItemData` 绑定到具体格位。
  - 说明右侧 Loot 是核心交互主板，不是简单装饰图。
- `BattleIntelligence_Main.cs` / `BattleIntelligence2_Main.cs` / `BattleIntelligence_Main3.cs`
  - 根据品质、类型、轮廓、价格、占格数过滤 `Table_Item`。
  - `BattleIntelligence2_Main` 支持从一个已知/半已知格子进入筛选，锁定品质、类型或轮廓。
- `BattleRandom_Main.cs`
  - 随机拍卖/随机仓流程。
- `BattlePrevPanel_Main.cs`
  - 战前选择与地图/角色流程。

对复刻的意义：

- 局内应采用三栏高密度 HUD：左侧玩家轨迹，中间情报流与出价反馈，右侧空间化 Loot。
- “实时情报”不是图鉴展示，而是从当前格子线索反查候选藏品价值的估价工具。
- 出价不是普通输入框，应该有独立键盘、确认、倍率、上一轮报价和风险提示。

## 二、配置与数值结构

### 1. 已提取配置

已解码 52 张表，其中关键表：

- `BidMap.txt`：125 行，仓型、竞拍人数、成交线、掉落组、随机技能池。
- `Item.txt`：1136 行，藏品名、品质、基础价值、拍卖底价、占格形状、图标、模型。
- `Drop.txt`：629 行，仓型到藏品池的掉落关系。
- `Hero.txt`：20 行，角色名、职业、背景、技能、语音、BGM。
- `Skill.txt`：256 行，技能组、技能文本、目标、计数、触发与冷却。
- `RankMap.txt`：包含轮次时间、角色出生/配置、最低出价范围、竞拍类型等信息。

### 2. 成交线规则

`BidMap.auction_rounds_rate` 可读出样例：

```text
[2000, 1600, 1300, 1100, 0]
```

它对应 UI 口径：

- 第 1 轮：最高价达到第二名 200% 可成交。
- 第 2 轮：最高价达到第二名 160% 可成交。
- 第 3 轮：最高价达到第二名 130% 可成交。
- 第 4 轮：最高价达到第二名 110% 可成交。
- 第 5 轮：最高价高于第二名即可成交；同价加赛。

当前 `match-core` 的实现用“领先第二名的比例”表达：

- 100%、60%、40%、20%、0%。

两种口径数学等价，但 UI 文案必须统一成玩家更容易理解的 BidKing 口径：“达到第二名的 200%/160%/130%/110%/高于即可”。

### 3. 藏品格信息

BidKing 的 `Item` 表中关键字段：

- `slot_type`：藏品形状。
- `item_quality`：品质，对应白/绿/蓝/紫/金/红等稀有度边框。
- `base_value` / `auction_baseprice`：价值判断基础。
- `grid_count`：占格数量。
- `icon_path` / `icon_atlas` / `model_3D`：图标、图集、模型引用。

我们当前配置已具备：

- `rarity`
- `value`
- `displayValue`
- `footprint`
- `category`
- `iconKey`
- `repairCost`
- `setId`

需要强化的是：把 `footprint + rarity + category + valueRange` 用作右侧格子和实时情报的主要交互语言，而不是只在结算时显示。

## 四、当前《珍宝局》实现状态

### 1. 已完成的核心骨架

当前工程已经具备：

- 4 人房间，AI 补位。
- `coreMode`：同一仓 5 轮渐进揭示。
- 明拍/暗拍选择。
- 随机藏宝仓抽选 overlay。
- 掌眼人情报卡 overlay。
- 右侧仓库格。
- 出价键盘与二次确认。
- 每轮反馈、成交线、排名。
- 最终原地开箱、真值、利润、复盘。
- 图鉴、任务、段位、收藏柜、后台回放雏形。

### 2. 当前差距

与 BidKing 参考帧相比，主要差距是体验密度：

- 左侧玩家轨迹还不够像“5 轮读人面板”，空轮次槽位不稳定。
- 中央信息流偏工具面板，需要更像“拍卖师/角色情报日志”。
- 右侧 Loot 格子已经存在，但点击格子后没有直接进入“按该格线索锁定的实时情报”。
- 实时情报面板仍偏表单，应改成品质色块、类别、轮廓和候选价值列表的高频估价工具。
- 底部操作条已存在，但需要更接近 BidKing：出价、停手、技能、情报、表情始终作为主要操作区。
- 成交线 UI 文案需要统一成 BidKing 的比例口径。

## 六、本轮实装设计

### P0：局内 UI 与交互

1. 左侧玩家席位固定 5 个轮次槽：
   - 未出价显示轮次编号。
   - 已提交显示亮色点。
   - 已有排名显示排名。
   - 当前玩家高亮。
2. 右侧仓库格支持点击：
   - 点击未知/半已知格打开实时情报。
   - 如果格子已知品质、类别或轮廓，则自动锁定对应筛选。
   - 这对应 BidKing 中从格子反查候选藏品的行为。
3. 实时情报面板升级：
   - 顶部显示当前仓与被查看格。
   - 品质用色块按钮。
   - 类别用紧凑按钮。
   - 轮廓用形状 chip。
   - 列表显示候选藏品、类别、品质、形状、参考价值。
   - 顶部计算候选数量和价值区间。
4. 中央信息流强化：
   - 公共情报、私人情报、技能情报分来源展示。
   - 推荐上限只作为辅助，不抢主叙事。
5. 成交线口径统一：
   - UI 显示“达到第二名 200%/160%/130%/110%/高于即可”。
   - 内部仍可用领先比例计算。

### P1：流程感

1. 首轮进入时保留随机藏宝仓抽选。
2. 抽选后进入掌眼人情报卡。
3. 每轮中心信息流追加信息。
4. 最终成交后右侧格子原地逐个翻开，收益逐步滚动。

### P2：包装保持

1. 背景继续使用古代拍卖厅、宫廷旧藏、商船、军库等资产。
2. 角色继续使用三国/古代职业包装。
3. 藏品名称保留玉器、兵器、印章、古籍、香囊、军旗等古风语义。
4. 不引入现代商店/充值/交易黑箱。

## 七、验收标准

本轮完成后，需要能验证：

1. 创建房间进入对局后，流程为：随机仓 -> 掌眼人情报 -> 出价 -> 轮次反馈 -> 继续同仓 -> 最终开箱。
2. 明拍与暗拍都能完成 5 轮或提前成交。
3. 左侧 4 名玩家均有 1-5 轮轨迹。
4. 右侧 Loot 格子可点击打开实时情报，并自动带入品质/类别/轮廓筛选。
5. 出价键盘、二次确认、停手、技能、情报入口都能操作。
6. 最终结算在同一个仓库格中逐件揭示藏品并滚动利润。
7. 页面在桌面与移动宽度下不出现文字重叠。
8. `bitkingdom/reverse/` 和 `bitkingdom/doc/videos/` 不进入提交。
