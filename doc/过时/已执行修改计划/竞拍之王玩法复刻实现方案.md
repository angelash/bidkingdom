# 《竞拍之王》玩法复刻实现方案

> 口径：基于你上传的视频关键帧、角色教学视频、Steam 页面与公开攻略信息整理。以下不是照搬代码，而是面向项目落地的功能拆解、界面结构、操作流程、数据结构和实现细节。

> 当前修正口径：本方案优先 100% 复刻核心玩法，不把市场、交易、商城、活动、仓型皮肤等周边系统列为当前核心。风格、包装、仓型名称和题材可以替换，但核心规则必须先落准：**同一个仓库进行 5 轮渐进揭示，玩家结合每轮出价反馈、角色技能和可见信息推断最终合理竞拍价；竞拍模式分为明拍和暗拍两类。**

---

## 1. 产品定位

《竞拍之王》的核心不是传统拍卖，也不是局外收藏市场，而是“单仓五轮渐进揭示 + 信息不完全 + 多人出价推断 + 藏品估值 + 角色技能 + 心理博弈”。玩家不是在单纯比谁出价高，而是在同一个仓库真实价值不确定、对手判断不透明、技能信息不对称的情况下，连续 5 轮修正自己的估值和报价。

如果复刻，最小可行版本应该抓住四个核心：

1. 每局 4 人参与，围绕同一个仓库/集装箱进行 5 轮竞拍。
2. 仓库内藏品真实价值隐藏，5 轮中逐步揭示轮廓、稀有度、品类、数量、均值或价值区间。
3. 每个角色拥有不同的信息技能，改变自己对同一仓库的估值方式。
4. 竞拍模式分为明拍和暗拍：明拍重在公开压价和观察行为，暗拍重在隐藏报价和轮间反馈推断。
5. 最终赢家买下整个仓库并开箱结算，按“仓库总真实价值 - 成交价”计算收益。

不建议一开始把重点放在复杂 3D 场景、大量局外系统或交易市场上。这个品类真正的上头点来自“我是不是通过 5 轮信息看懂了这个仓库”“别人是不是误判了价格”“我最后的成交价是否压在合理区间”。所以第一版只优先保证局内信息流、五轮出价节奏、明拍/暗拍反馈、技能推理、开箱验证和复盘可读性。

## 2. 核心玩法循环

### 2.1 单局主循环

一局可以按如下流程实现：

1. 匹配 / 组队进入房间。
2. 选择角色。
3. 系统生成本局唯一仓库：场景背景、藏品列表、藏品布局、稀有度、真实价格、干扰物。
4. 展示仓库初始可见信息：部分轮廓、部分品质、公共价值提示或仓库描述。
5. 进入第 1 轮竞拍，玩家根据初始信息输入报价。
6. 回合结束后，系统按明拍/暗拍规则给出反馈：当前最高、排名、差距区间、领先状态或公开价格变化。
7. 进入下一轮，同一个仓库继续揭示新信息，角色技能也在指定轮次触发或由玩家主动使用。
8. 玩家结合“新增揭示 + 上轮出价反馈 + 对手行为 + 自身技能”修正估值并再次报价。
9. 重复 5 轮后，最终最高有效报价者获得整个仓库。
10. 开箱逐件揭示仓库全部藏品，计算总真实价值、成交价、盈利/亏损、排名、奖励。
11. 进入复盘页：展示每轮出价、每轮新增信息、关键技能、仓库真实价值和玩家误判点。

### 2.2 回合数量

建议默认 5 轮。公开视频与公开攻略中也提到“五轮竞价”的结构。5 轮的好处是：

- 第 1 轮：试探。
- 第 2 轮：技能开始提供方向。
- 第 3 轮：信息差扩大。
- 第 4 轮：进入真假博弈。
- 第 5 轮：最终决策。

如果回合太少，技能没有发挥空间；如果回合太多，节奏会拖，玩家会疲劳。

### 2.3 五轮渐进揭示节奏

五轮必须围绕同一个仓库逐步增加信息，而不是每轮换一个新仓库。推荐基础节奏如下：

| 轮次 | 揭示目标 | 玩家主要判断 |
|---|---|---|
| 第 1 轮 | 仓库主题、部分轮廓、粗估值或少量品质 | 这个仓库大概值不值得进 |
| 第 2 轮 | 随机若干藏品品质、类别数量或平均值 | 仓库价值下限是否可靠 |
| 第 3 轮 | 技能信息开始形成差异，部分关键格子被标记 | 自己和对手可能看到了什么 |
| 第 4 轮 | 高价值/陷阱信号更明显，报价差距开始收敛 | 是否继续抬价或诱导别人接盘 |
| 第 5 轮 | 最后一批关键线索或最终状态反馈 | 给出最终成交价，决定是否搏 |

每轮的揭示不能直接公布真实总价，只能增加“可推理信息”。玩家的核心乐趣来自在第 1-5 轮之间持续修正自己的心理价。

### 2.4 胜负目标

单局目标不是“拍到仓库”本身，而是“用低于真实价值的价格拍到仓库”或“让别人高价拍到低价值仓库”。

核心公式：

```
收益 = 仓库真实总价值 - 最终成交价
```

如果没有拍到仓库，收益可以为 0，也可以根据模式设计为少量观战奖励 / 排名积分。第一版建议未拍到者不扣钱，只按局内表现给少量经验，避免挫败过强。

### 2.5 明拍与暗拍两种核心模式

首版核心只保留两类竞拍模式：

| 模式 | 核心体验 | 信息反馈 |
|---|---|---|
| 明拍 | 公开抬价、观察谁愿意继续跟、用行为压力逼对手犯错 | 当前领先价、领先者、放弃状态、公开加价过程 |
| 暗拍 | 隐藏报价、根据轮间排名/差距反馈推断对手心理价 | 已出价状态、轮后排名、相对差距、是否领先或接近领先 |

其他包装式仓型、活动入口、局外市场都不能替代这两个模式。当前复刻重点是把同一个仓库在明拍和暗拍下的五轮推价逻辑做准确。

## 3. 局内界面结构

根据视频，局内界面可以拆成以下区域。

### 3.1 主界面布局

推荐横屏结构：

```
┌──────────────────────────────────────────┐
│ 顶部：回合数 / 倒计时 / 当前资金 / 设置       │
├───────────────┬──────────────────────────┤
│ 左侧玩家列表    │ 中部：系统消息 / 技能消息 / 出价日志 │
│ 头像、角色、出价 │                            │
│ 状态、排序反馈   │                            │
├───────────────┼──────────────────────────┤
│ 左下：数字报价盘 │ 右侧：仓库/藏品布局图           │
│ 快捷倍率/确认    │ 藏品轮廓、颜色、稀有度、已知线索   │
└───────────────┴──────────────────────────┘
```

### 3.2 左侧玩家列表

展示内容：

- 玩家头像。
- 玩家昵称。
- 使用角色头像。
- 当前回合出价状态：未出价 / 已出价 / 放弃。
- 上一轮排序：第 1、第 2、第 3、第 4。
- 可选：表情/气泡，用于心理博弈。

暗拍模式下，不能直接显示具体金额，但可以显示“已出价”“领先”“接近领先”“明显落后”等模糊状态。

### 3.3 中部消息区

这是游戏的信息核心，需要非常清楚。建议按类型分颜色：

- 系统信息：本轮开始、倒计时、出价完成。
- 仓库信息：仓库总价值区间、某类藏品数量、某品质数量。
- 技能信息：谁的技能触发、揭示了什么。
- 对手行为：某玩家连续加价、某玩家放弃、某玩家可能领先。
- 风险提示：当前报价可能高于已知估值、报价过低可能拍不到。

所有消息要能回看。建议做“消息历史折叠面板”，因为这个游戏的信息记忆成本很高。

### 3.4 右侧仓库布局图

视频里仓库藏品以不同尺寸矩形/轮廓分布在仓库区域中，颜色代表品质或信息状态。建议第一版不做复杂 3D 交互，直接做 2D 平面布局。

藏品格子显示状态：

- 未知：灰色轮廓。
- 已知尺寸：显示矩形大小。
- 已知品质：显示品质颜色。
- 已知价值：显示价格标签。
- 已被技能标记：加技能图标或描边。
- 疑似高价值：加发光边框。
- 疑似陷阱：加警告标记。

品质颜色建议：

- 白：普通。
- 绿：良好。
- 蓝：稀有。
- 紫：史诗。
- 橙 / 金：传说。
- 红：极品 / 特殊。

### 3.5 左下报价盘

视频中能看到数字键盘和倍率按钮。建议保留这种结构，因为它很适合快速输入。

报价盘功能：

- 数字键 0-9。
- 删除键。
- 00 快捷键。
- 当前输入金额。
- 快捷倍率：x1.2、x1.4、x2.0 等，基于当前领先价或系统参考价自动填入。
- 确认出价。
- 放弃 / 跳过。
- 风险提示：预计利润、超过已知估值、报价低于竞争区。

第一版必须做“二次确认”，尤其是高额报价时，避免误触导致体验崩坏。

## 4. 玩家操作流程

### 4.1 进入游戏

主界面入口：

- 快速匹配。
- 好友房。
- 角色。
- 藏品图鉴。
- 战绩。
- 教程。
- 商城 / 外观。

第一版建议只做快速匹配 + 角色选择 + 局内 + 结算。好友房可以后置。

### 4.2 角色选择

角色选择界面需要展示：

- 角色立绘。
- 技能名称。
- 技能效果。
- 上手难度。
- 定位标签。
- 推荐打法。

角色标签示例：

- 新手稳定。
- 低风险。
- 高信息量。
- 估值型。
- 干扰型。
- 后期型。
- 心理博弈型。

角色选择不应只展示文案，要展示技能会在仓库界面产生什么视觉效果。比如“揭示 2 个未知品质”就直接在小预览图上高亮两个格子。

### 4.3 局内出价

每一轮操作：

1. 查看仓库布局。
2. 查看角色技能揭示的信息。
3. 查看系统消息和上一轮反馈。
4. 估算仓库价值区间。
5. 输入报价。
6. 确认。
7. 等待其他玩家。
8. 回合结算反馈。

倒计时建议 45-60 秒。新手局可以 60 秒，熟练局 35-45 秒。

### 4.4 出价反馈

回合结束后不要直接公布全部报价。建议公布以下信息：

- 当前最高出价者的身份，但不显示具体金额；或只显示“某玩家暂时领先”。
- 所有玩家的排名顺序。
- 与领先价的相对差距：接近 / 略低 / 明显低。
- 自己报价是否有效。
- 自己报价是否超过当前已知估值区间。

可选反馈：

- “你的报价位于全场第 2”。
- “你的报价接近当前最高报价”。
- “你的报价明显低于主流区间”。
- “当前竞争较激烈”。
- “至少 2 名玩家提高了报价”。

### 4.5 最终开箱

开箱阶段是情绪峰值，必须做足反馈。

展示顺序：

1. 仓库门打开。
2. 逐件揭示藏品。
3. 每件显示：品质、名称、真实价值、是否被玩家技能提前识别。
4. 累加总价值。
5. 显示成交价。
6. 显示最终盈亏。
7. 播放大赚 / 小赚 / 持平 / 血亏表现。
8. 展示其他玩家反应。
9. 排名与奖励。

不要只给一个最终数字。这个游戏的爽点是“翻牌式揭示”和“逐步验证自己的判断”。

## 5. 藏品与仓库生成

### 5.1 仓库数据结构

建议服务端生成完整真实数据，客户端只拿到自己可见的信息。

```ts
type Warehouse = {
  warehouseId: string;
  sceneType: string;
  seed: number;
  items: AuctionItem[];
  totalValue: number;
  baseEstimate: ValueRange;
  difficulty: number;
};

type AuctionItem = {
  itemId: string;
  templateId: string;
  name: string;
  category: string;
  rarity: Rarity;
  value: number;
  size: { w: number; h: number };
  position: { x: number; y: number };
  visibleShape: boolean;
  visibleRarity: boolean;
  visibleValue: boolean;
  tags: string[];
};

type ValueRange = {
  min: number;
  max: number;
};
```

### 5.2 藏品模板

藏品模板字段：

```ts
type ItemTemplate = {
  templateId: string;
  name: string;
  category: string;
  rarityWeights: Record<Rarity, number>;
  baseValueRange: ValueRange;
  sizeRange: { minW: number; maxW: number; minH: number; maxH: number };
  visualAsset: string;
  silhouetteAsset: string;
  description: string;
};
```

### 5.3 价值生成

不要纯随机。要保证每局有“可推理性”。推荐三层生成：

1. 先决定仓库总价值档位：低价值、中价值、高价值、陷阱仓、极品仓。
2. 再决定品质分布：白绿蓝紫金数量。
3. 最后生成具体藏品价格。

示例：

```ts
WarehouseTier = Low | Normal | High | Trap | Jackpot
```

每个档位配置：

```json
{
  "Normal": {
    "totalValueMin": 80000,
    "totalValueMax": 180000,
    "itemCountMin": 8,
    "itemCountMax": 16,
    "rarityWeight": {
      "white": 45,
      "green": 30,
      "blue": 18,
      "purple": 6,
      "gold": 1
    }
  }
}
```

### 5.4 仓库布局

布局要服务玩法，而不是只服务美术。矩形大小、遮挡、颜色和位置都应提供线索。

建议：

- 大件不一定高价值，但有更高概率高价值。
- 小件可能是高价值古董/首饰。
- 同类藏品可形成套装价值。
- 某些区域可被技能扫描。
- 某些藏品只显示轮廓，不显示品质。

布局算法：

1. 根据仓库尺寸生成二维网格。
2. 按物品尺寸从大到小放置。
3. 避免重叠。
4. 部分允许视觉遮挡，但逻辑上仍可点选。
5. 根据技能和回合决定揭示层级。

## 6. 信息可见性系统

这个游戏的核心系统不是“拍卖”，而是“信息权限”。建议所有信息都走统一可见性层。

### 6.1 信息类型

每个藏品有以下信息：

- 是否存在。
- 轮廓。
- 尺寸。
- 品质。
- 类别。
- 价值区间。
- 精确价值。
- 是否为套装。
- 是否为假货 / 低价值陷阱。

### 6.2 可见性结构

```ts
type PlayerItemKnowledge = {
  playerId: string;
  itemId: string;
  knowsExistence: boolean;
  knowsShape: boolean;
  knowsRarity: boolean;
  knowsCategory: boolean;
  knowsValueRange?: ValueRange;
  knowsExactValue?: number;
  source: KnowledgeSource[];
};

type KnowledgeSource =
  | "InitialReveal"
  | "RoundReveal"
  | "CharacterSkill"
  | "Prop"
  | "SystemHint"
  | "OpponentMessage";
```

### 6.3 信息推送原则

服务端只下发玩家可见的信息，避免客户端反编译直接拿到真实价值。

客户端显示仓库时，不应该拿完整 Warehouse，而应该拿：

```ts
type WarehouseViewForPlayer = {
  warehouseId: string;
  round: number;
  visibleItems: VisibleItem[];
  knownTotalRanges: ValueRange[];
  messages: AuctionMessage[];
};
```

## 7. 竞价系统

### 7.1 模式总则

竞价系统只围绕同一个仓库运行 5 轮。每轮玩家都在更新同一个最终成交价判断，不是拍 5 个不同标的。系统需要同时支持明拍和暗拍两种模式，二者共用仓库、知识、技能和结算结构，只改变出价过程与反馈方式。

```ts
type Bid = {
  matchId: string;
  round: number;
  playerId: string;
  amount: number;
  mode: "Open" | "Sealed";
  status: "Submitted" | "Skipped" | "Timeout";
  submitTime: number;
};
```

### 7.2 暗拍模式

暗拍模式下，每轮玩家输入一个报价，回合结束前互相不可见。它的关键不是“谁最后点得快”，而是玩家通过 5 轮反馈逐步推断对手心理价。

回合中显示：

- 谁已经提交报价。
- 谁还在犹豫。
- 倒计时。
- 自己当前输入价和风险提示。

回合结束后显示：

- 本轮报价排名。
- 自己与最高价的相对差距：接近 / 略低 / 明显低。
- 当前领先玩家，或只显示“有人领先”。
- 可选：价格区间反馈，例如最高价落在 80,000～100,000。
- 不直接公布所有具体报价，除非进入最终复盘。

暗拍的 5 轮报价可以允许上下调整。玩家可以前期试探、后期回撤，也可以故意出高价制造行为假象；系统最终取第 5 轮有效报价成交。

### 7.3 明拍模式

明拍模式下，当前价、领先者、放弃状态公开。它的核心是公开压力、跟价节奏和行为观察。

回合中显示：

- 当前最高价。
- 当前领先玩家。
- 每名玩家是否仍在场。
- 公开加价记录。
- 倒计时和最低加价单位。

明拍每轮仍然围绕同一个仓库进行。每一轮新增仓库信息后，明拍价格从上一轮结束价或本轮起拍价继续推进，玩家可继续跟价或放弃。回合结束后，系统进入下一轮揭示，而不是立刻结算仓库。

推荐规则：

- 每轮有独立倒计时。
- 玩家可加价或放弃。
- 放弃可设计为“本轮放弃”或“整局退出竞拍”，第一版建议只做本轮放弃，避免过早出局。
- 第 5 轮结束时，以最终公开最高价成交。

### 7.4 有效报价规则

建议规则：

- 报价必须大于 0。
- 报价不能超过玩家当前资金。
- 暗拍每轮可重新报价，最终取第 5 轮有效报价。
- 明拍报价必须大于当前最高价，并满足最低加价单位。
- 超时未报价视为本轮跳过或沿用上一轮报价，具体由模式配置决定。

从心理博弈角度，暗拍必须允许每轮重新报价，不强制递增。这样玩家可以第 1 轮虚高试探，第 2 轮回撤，也可以制造假象。明拍则保留递增加价，因为公开价格本身就是行为压力。

### 7.5 成交规则

默认第 5 轮结束后最高有效报价者成交，买下整个仓库。

平价处理：

1. 更早提交者胜。
2. 或触发加时一轮。
3. 或按角色 / 道具优先级。

第一版建议“更早提交者胜”，规则简单。

### 7.6 每轮反馈数据

每轮结束都要生成可复盘的反馈对象，这是 5 轮推理的关键：

```ts
type RoundBidFeedback = {
  round: number;
  mode: "Open" | "Sealed";
  leaderPlayerId?: string;
  selfRank?: number;
  selfGapToLeader?: "Close" | "Medium" | "Far";
  publicRanking?: Array<{ playerId: string; rank: number }>;
  publicPrice?: number;
  priceBand?: ValueRange;
  message: string;
};
```

暗拍用排名、差距和区间制造推断空间；明拍用公开价格、领先者和放弃行为制造压力。

### 7.7 资金规则

每位玩家初始资金相同，如 1,000,000。

可以设计为：

- 单局内资金：只用于本局报价。
- 赛季资产：累计收益。
- 排位积分：根据收益、胜负、名次计算。

不要让真实付费和局内报价强绑定，否则容易走向赌博化和付费不公平。

## 8. 角色技能设计

角色技能本质是改变信息获取方式，而不是直接加钱。这样最适合保持策略公平。

### 8.1 技能分类

1. 品质揭示型：显示某些藏品品质。
2. 价值区间型：显示某些藏品或仓库总价值区间。
3. 数量统计型：显示某品质数量、某类别数量。
4. 区域扫描型：揭示某个区域内藏品信息。
5. 轮廓预判型：显示部分隐藏物品轮廓。
6. 避坑型：识别低价值藏品或假货。
7. 干扰型：对其他玩家展示模糊/错误/延迟信息。
8. 后期爆发型：前几轮弱，后几轮信息量暴增。

### 8.2 已观察到的角色方向

根据视频与公开资料，可整理出以下角色设计方向，第一版不需要完全一致，可以做等价替代。

#### 索菲：渐进式信息型

定位：新手友好、稳定信息流。

视频里能看到“每回合自动揭示两个品质未知藏品的品质”的说明。

设计实现：

```ts
Skill_Sophie(roundStart) {
  const candidates = unknownRarityItems(player);
  const selected = randomPick(candidates, 2, seed);
  revealRarity(player, selected);
}
```

特点：

- 每回合都有收益。
- 不需要玩家主动选择。
- 适合新手形成估值路径。
- 缺点是信息分散，遇到大仓库时可能不够精准。

#### 玛利亚：避坑型

定位：低风险、过滤低价值陷阱。

视频里显示其技能会在开局揭示白色、绿色、蓝色等低品质藏品相关价值/品质信息，属于“风险前置控制”。

设计实现：

```ts
Skill_Maria(matchStart) {
  const lowItems = items.filter(i => ["white", "green", "blue"].includes(i.rarity));
  revealRarity(player, lowItems);
  revealTotalValue(player, lowItems);
}
```

特点：

- 快速判断“这个仓库是不是垃圾多”。
- 非常适合新手避免高价接盘。
- 峰值收益不一定最高，但下限高。

#### 陈美：仓库估值型

定位：整体估值、快速判断仓库是否值得争。

视频标题和画面都指向“仓库估值神器”，核心应是围绕整个仓库总价值或均价提供信息。

设计实现：

```ts
Skill_ChenMei(matchStartOrRoundStart) {
  const range = calculateWarehouseValueRange(warehouse, accuracyByRound(round));
  revealWarehouseValueRange(player, range);
}
```

可设计成：

- 第 1 轮显示一个较宽的总价值区间。
- 后续每轮区间逐步收窄。
- 或直接显示“当前仓库预估总价 = 真实总价 ± 20%”。

特点：

- 上手直观。
- 强在宏观判断，弱在不知道具体哪个藏品值钱。
- 容易被对手观察报价行为后跟随。

#### 吴起灵：古玩鉴定型

定位：识别高价值古玩 / 特定类别。

视频中出现“古玩鉴定高手”，技能方向应围绕古玩类藏品的品质、数量或价值。

设计实现：

```ts
Skill_WuQiling(matchStart) {
  const antiqueItems = items.filter(i => i.category === "Antique");
  revealCount(player, "Antique", antiqueItems.length);
  revealBestItemRarityOrRange(player, antiqueItems);
}
```

特点：

- 面对古玩仓强势。
- 面对非古玩仓偏弱。
- 适合做“高上限、吃仓库类型”的角色。

#### 伊森：轮廓预判型

定位：根据轮廓/形状判断仓库价值。

视频标题是“轮廓预判大师”，画面中也有大量仓库格子和轮廓信息。

设计实现：

```ts
Skill_Ethan(matchStart) {
  const hiddenItems = unknownShapeItems(player);
  const selected = pickByValuePotential(hiddenItems, count = 4);
  revealShape(player, selected);
}
```

可进一步设计：

- 开局显示若干高潜力藏品轮廓。
- 但不显示品质和价值。
- 玩家要根据尺寸、位置、类别进行推理。

特点：

- 熟练玩家收益高。
- 新手可能看不懂。
- 适合做中高阶角色。

#### 加布里埃拉：新手福音型

定位：直接、稳定、低理解成本。

公开视频和攻略中常把她描述为上手简单、稳定获取信息。

设计实现：

- 每回合随机显示 1-2 件藏品的价值区间。
- 或显示当前仓库中某一件高价值藏品的品质。

特点：

- 新手拿到信息就能用。
- 不需要复杂推理。
- 上限不能太高，否则会压制其他角色。

## 9. 道具系统

道具是局内二级决策，不属于当前 100% 复刻的核心。当前核心必须先把“角色技能 + 5 轮揭示 + 明拍/暗拍出价反馈”做准，道具只作为后续包装和扩展。

后续如果要补，可考虑 4 种：

1. 单品扫描：查看一个藏品的品质。
2. 区域扫描：查看一个区域内 2-4 个藏品的轮廓或品质。
3. 估值报告：获得仓库总价值区间。
4. 干扰广播：向所有人发送一条系统风格的模糊提示，但不能伪造官方结论。

道具必须有限制：

- 每局最多使用 1-2 次。
- 消耗局内道具点，不直接用真实付费购买局内优势。
- 使用后写入消息区，其他玩家知道“有人用了道具”，但不知道具体获得了什么。

当前核心版本不依赖道具成立，因此开发优先级低于单仓五轮、两种竞拍模式和角色技能。

## 10. AI 对手 / 机器人

如果要快速验证玩法，机器人很关键。机器人不能只随机报价，否则这个游戏很快失去博弈感。

### 10.1 机器人状态

```ts
type BotState = {
  riskPreference: number; // 0-1
  confidence: number;     // 对仓库估值的信心
  estimatedValue: ValueRange;
  targetProfitRate: number;
  bluffLevel: number;
  aggression: number;
};
```

### 10.2 机器人报价公式

```ts
botBid = estimatedMeanValue * (1 - targetProfitRate) * aggressionModifier * confidenceModifier
```

示例：

- 保守型：目标利润 35%，报价偏低。
- 激进型：目标利润 10%，容易抢仓。
- 诈唬型：前几轮高报，最后回撤。
- 跟随型：根据玩家报价排序反馈贴近领先区。

### 10.3 机器人行为表现

机器人需要表现出“像人在判断”：

- 倒计时末尾才出价。
- 连续两轮加价。
- 某轮突然放弃。
- 使用表情。
- 看到高价值线索后更激进。

但不能让机器人读全量真实数据。机器人也应该使用模拟信息视野，否则玩家会觉得被系统作弊。

## 11. 匹配与模式

### 11.1 模式划分

第一版建议：

1. 新手教学：单人 + 机器人，引导完整流程。
2. 快速匹配：4 人暗标。
3. 好友房：可邀请好友，适合传播。
4. 练习模式：自选角色 + 自选仓库难度。

### 11.2 新手保护

前 3-5 局建议强制机器人局或半机器人局，并使用固定仓库模板。

原因：

- 玩家第一次玩时不知道如何估值。
- 如果第一局被老玩家诱导血亏，流失会很高。
- 教程要让玩家体验一次“低价捡漏”和一次“高价接盘”的差异。

### 11.3 教程设计

教程不要长篇文案，按局内步骤做弱引导：

1. 点开仓库，看藏品颜色。
2. 使用角色技能。
3. 根据提示估算总价值。
4. 输入一个推荐报价。
5. 看出价排序。
6. 第二轮调整报价。
7. 最终开箱验证。

关键是让玩家明白：

- 不是所有仓库都要抢。
- 出价不是越高越好。
- 技能信息只是不完整线索。
- 对手行为也可以成为线索。

## 12. 结算与成长

### 12.1 单局结算

结算页字段：

- 仓库真实总价值。
- 最终成交价。
- 最终收益。
- ROI：收益 / 成交价。
- 拍得玩家。
- 每位玩家最终报价。
- 每轮报价曲线。
- 技能贡献：本局通过技能提前识别了多少价值。
- 最佳判断：谁的最终报价最接近合理价。

### 12.2 排名规则

可以按收益排名，而不是只按是否拍到仓库。

建议：

- 拍到并盈利：高分。
- 拍到但亏损：低分甚至扣分。
- 没拍到但成功诱导别人亏损：可给“心理博弈分”。
- 报价最接近真实合理成交价：给“估值分”。

### 12.3 长线成长

适合的长线系统：

- 角色解锁。
- 角色熟练度。
- 藏品图鉴。
- 称号。
- 表情。
- 头像框。
- 仓库主题皮肤。
- 排位段位。

不建议做数值养成增强局内优势。这个游戏的公平感很重要，一旦付费角色明显更强，会直接破坏心理博弈基础。

## 13. UI/UX 细节

### 13.1 信息可读性优先

这个游戏信息量大，UI 不能炫技过度。必须保证：

- 已知信息和未知信息区分明显。
- 技能新增信息要有动画高亮。
- 消息区支持筛选：全部 / 我的信息 / 技能 / 出价。
- 重要提示不能被弹幕、主播框、活动入口遮挡。

### 13.2 报价风险提示

玩家输入报价后，实时显示：

- 当前报价。
- 已知最低估值。
- 已知最高估值。
- 预计利润区间。
- 风险等级。

示例：

```
当前报价：120,000
已知估值：90,000 - 180,000
预计收益：-30,000 ~ +60,000
风险：中
```

这样能降低新手挫败。

### 13.3 复盘必须强

玩家输掉后，如果不知道为什么输，会骂随机、黑箱、机器人作弊。复盘页必须告诉他：

- 哪些信息其实已经暗示仓库很值钱。
- 哪些信息是误导。
- 对手为什么敢加价。
- 自己在哪一轮报价过高或过低。

可以做“AI 复盘点评”：

- “你第 3 轮已经知道蓝色藏品数量较多，但没有提高报价。”
- “你在总价值区间未确认前使用 x2 报价，风险过高。”
- “对手第 4 轮突然放弃，说明他可能识别到低价值陷阱。”

## 14. 技术实现建议

### 14.1 前后端分工

服务端权威：

- 生成仓库。
- 保存真实藏品价值。
- 控制信息可见性。
- 接收报价。
- 结算胜负。
- 防作弊。

客户端负责：

- 表现仓库。
- 输入报价。
- 播放技能揭示动画。
- 展示消息。
- 开箱表现。

### 14.2 服务端核心模块

```txt
MatchService
- 创建房间
- 匹配玩家
- 管理回合状态

WarehouseService
- 根据 seed 生成仓库
- 生成藏品价值和布局

KnowledgeService
- 计算每个玩家可见信息
- 处理技能揭示
- 处理道具揭示

BidService
- 接收报价
- 校验报价
- 计算回合反馈

SettlementService
- 最终成交
- 开箱结果
- 收益和积分

ReplayService
- 保存每轮报价
- 保存技能触发
- 保存最终仓库真实数据
```

### 14.3 状态机

```ts
enum MatchState {
  WaitingPlayers,
  CharacterSelect,
  WarehousePreview,
  RoundStart,
  Bidding,
  RoundReveal,
  FinalReveal,
  Settlement,
  Finished
}
```

### 14.4 网络同步

每轮只需要同步关键事件，不需要帧同步。

事件示例：

```ts
type MatchEvent =
  | { type: "RoundStarted"; round: number; deadline: number }
  | { type: "KnowledgeUpdated"; playerId: string; updates: VisibleItemUpdate[] }
  | { type: "BidSubmitted"; playerId: string }
  | { type: "RoundResult"; ranking: BidRankInfo[]; hints: AuctionMessage[] }
  | { type: "FinalWinner"; playerId: string; amount: number }
  | { type: "ItemRevealed"; itemId: string; value: number; rarity: Rarity }
  | { type: "Settlement"; results: PlayerResult[] };
```

### 14.5 防作弊

重点：不要把真实仓库价值发给客户端。

客户端只拿可见信息。即使本地抓包，也只能看到当前玩家该知道的内容。

服务端要校验：

- 报价是否超资金。
- 报价是否在时间内。
- 技能是否能触发。
- 道具是否还有次数。
- 玩家是否重复提交。

## 16. 平衡设计

### 16.1 角色强度原则

角色不能简单分强弱，而要分信息维度。

- 新手角色：信息直接，但上限低。
- 熟练角色：信息间接，但上限高。
- 后期角色：前期弱，后期强。
- 特化角色：遇到特定仓库强，泛用性弱。

### 16.2 信息价值量化

可以给每种信息定价值：

- 显示一个白色品质：1 分。
- 显示一个蓝色品质：3 分。
- 显示一个紫色品质：5 分。
- 显示一个精确价值：8 分。
- 显示仓库总价值宽区间：6 分。
- 显示仓库总价值窄区间：12 分。

每个角色每局总信息价值应接近，但形式不同。

### 16.3 仓库价值分布

避免极端随机导致玩家觉得“全靠命”。建议：

- 70% 普通局：可通过信息推理出合理报价。
- 20% 高波动局：存在大赚/大亏可能。
- 10% 特殊局：极品仓或陷阱仓，制造传播素材。

### 16.4 报价合理区间

系统内部可以计算一个“理性成交价”：

```ts
fairBid = expectedValue * (1 - desiredProfitRate)
```

玩家最终报价与 fairBid 的偏差可用于结算评分和 AI 复盘。

## 17. 玩家反馈问题与改进方向

公开页面显示游戏评价并不稳定，说明这个题材有热度，但也容易被骂。复刻时要重点规避以下问题。

### 17.1 问题：随机感过强

解决：

- 提供明确复盘。
- 增加可推理线索。
- 控制极端仓比例。
- 新手局使用教学仓库。

### 17.2 问题：信息太乱，看不懂

解决：

- 消息分类。
- 技能信息高亮。
- 估值面板实时汇总。
- 鼠标/触屏点选藏品显示当前已知信息。

### 17.3 问题：角色强度差异大

解决：

- 按信息价值量化平衡。
- 做角色胜率、平均收益、使用率统计。
- 免费角色必须有稳定可玩性。
- 付费/解锁角色不能明显数值碾压。

### 17.4 问题：被机器人/对手坑了但不知道原因

解决：

- 展示每轮报价曲线。
- 展示对手已知信息推测。
- 展示关键误判节点。
- 对机器人标记为机器人，避免信任问题。

### 17.5 问题：美术廉价感

解决：

- 不追求大量精细物件，先统一视觉风格。
- 仓库背景可用高质量 2D 图。
- 藏品用卡片/图标化表达，保证清晰。
- 关键开箱藏品再做高质量图。

## 19. 小程序/手游化改造建议

如果目标是微信小程序或移动端，不建议照搬 PC 横屏复杂信息密度。建议做简化：

### 19.1 操作简化

- 报价用滑条 + 数字输入双模式。
- 快捷按钮：保守价 / 合理价 / 激进价。
- 估值面板自动汇总，减少玩家手算。
- 仓库图支持点击放大。

### 19.2 时长压缩

PC 一局可以 8-10 分钟，移动端建议 4-6 分钟。

方案：

- 保留 5 轮，但缩短每轮时间。
- 每轮 30-40 秒。
- 开箱可跳过，但保留关键动画。

### 19.3 社交传播

适合做：

- “我 3 万拍到 30 万仓库”的战报分享。
- “我让朋友 20 万接盘废仓”的反向炫耀。
- 结算图一键分享。
- 好友房观战。

## 20. 关键体验判断标准

复刻是否成功，不看功能数量，而看以下几个指标：

1. 玩家第 1 局是否能理解“低价买高值仓”的目标。
2. 玩家第 2 局是否开始主动看技能信息。
3. 玩家第 3 局是否会根据对手行为调整报价。
4. 输了以后是否觉得“我下次能判断更准”，而不是“纯随机”。
5. 赢了以后是否愿意截图分享。

如果这 5 点成立，这个玩法就有继续扩展价值。

## 22. 最终建议

这个玩法最值得复刻的不是“竞拍”外壳，而是“低成本制造多人信息差”的结构。它的研发成本可控，传播性强，但成败高度依赖信息可读性和复盘可信度。

第一版要少做复杂系统，多做三件事：

1. 让玩家能看懂自己为什么报价。
2. 让玩家能看懂别人为什么亏。
3. 让玩家输了也觉得是自己判断失误，而不是系统随机坑他。

只要这三点成立，后面再加角色、仓库、道具、排位、好友房，都会比较顺。

## 23. 修正补充：核心玩法 100% 复刻口径

> 修正口径：仓型、场馆、题材、美术风格属于包装层，不是当前核心。当前阶段不把市场、交易、商城、活动、角色外观、道具商店等周边系统列入 P0。P0 只有一件事：把“同一仓库 5 轮渐进揭示，玩家根据明拍/暗拍出价反馈和角色技能推断最终合理成交价”做准。

### 23.1 当前必须改正的理解

之前把“仓型/场馆选择、市场交易、局前装备”提得过高，会稀释核心。正确优先级应该是：

```text
核心玩法层：
单仓生成
→ 5轮渐进揭示
→ 明拍/暗拍出价
→ 每轮出价反馈
→ 角色技能改变信息视野
→ 玩家修正心理价
→ 第5轮最终成交
→ 开箱验证和复盘

包装层：
仓型名称、场馆背景、题材风格、角色包装、藏品美术

周边层：
市场、交易、商城、活动、外观、道具商店、长期养成
```

当前开发与文档都必须先围绕核心玩法层展开。包装层只服务核心判断，周边层后置。

### 23.2 单仓五轮渐进揭示是第一原则

复刻目标不是“5 轮 5 个货柜”，而是“1 个仓库跑 5 轮”。同一仓库在每一轮逐步露出更多可推理信息。

```text
第1轮：只知道仓库包装、少量轮廓、粗估值或基础品类
第2轮：新增若干格子的稀有度/轮廓/类别
第3轮：角色技能开始拉开信息差
第4轮：高价值与陷阱信号进一步显现，对手报价行为变得关键
第5轮：最终判断，给出成交价
```

玩家每轮的心理活动应该是：

```text
我现在知道了什么？
对手可能知道了什么？
上一轮谁报得更高？
我的报价离领先价近不近？
新增信息是否支持继续抬价？
最终价格应该压在什么区间？
```

如果没有这种 5 轮连续修正心理价的过程，就不是核心复刻。

### 23.3 明拍和暗拍是两套核心体验

首版只需要两种拍卖模式：明拍、暗拍。

| 模式 | 必须复刻的体验 | 每轮反馈 |
|---|---|---|
| 明拍 | 公开价格压力、观察谁还敢跟、通过放弃/跟价判断对手信心 | 当前价、领先者、加价记录、放弃状态 |
| 暗拍 | 报价隐藏、通过轮后排序和差距推断对手心理价 | 已出价状态、排名、相对差距、价格区间或领先提示 |

明拍不是另一个货柜，暗拍也不是单轮结算。两者都必须在同一仓库上跑完 5 轮，只是每轮出价和反馈方式不同。

### 23.4 角色技能只服务信息差

角色技能的核心作用是改变玩家对同一仓库的可见信息，不是直接给钱，也不是复杂养成。

首版角色技能应只覆盖这些信息维度：

- 揭示若干未知格子的稀有度。
- 揭示若干藏品轮廓。
- 揭示某类藏品数量。
- 揭示仓库总价值粗区间，并随轮次收窄。
- 标记可能低价值/陷阱藏品。
- 读取对手本轮可能报价区间。

技能结果必须进入当轮消息区和最终复盘。玩家需要知道“我为什么敢报这个价”，也需要知道“别人可能为什么敢跟价”。

### 23.5 界面只围绕核心信息布局

当前核心界面应围绕四个区域，不引入周边入口干扰：

```text
顶部：
  回合 1/5、模式、倒计时、当前资金

左侧：
  4名玩家、角色、出价状态、上一轮排名/放弃状态

中部：
  本轮系统消息、技能消息、出价反馈、风险提示

右侧：
  同一个仓库的格子化 Loot / 藏品布局
  轮廓、稀有度颜色、已知类别、已知价值区间、技能标记

底部：
  报价输入、快捷价、确认、放弃
```

包装差异可以体现在背景、仓库名、藏品美术和 UI 皮肤，但不能改变核心布局目标：让玩家一眼看到“仓库信息、对手状态、自己报价、轮次反馈”。

### 23.6 格子化仓库是核心界面，不是单纯开箱表现

Loot 格子不是局后奖励展示，而是五轮推理过程的一部分。每轮揭示都应直接作用在格子上：

```ts
type WarehouseSlotView = {
  slotId: string;
  itemId?: string;
  x: number;
  y: number;
  w: number;
  h: number;
  visibleShape: boolean;
  visibleRarity?: Rarity;
  visibleCategory?: string;
  visibleValueRange?: ValueRange;
  markedBySkill?: boolean;
  markReason?: string;
};
```

每轮开始时，服务端下发当前玩家可见的 `WarehouseSlotView[]`。客户端不能拿完整真实价值。不同玩家因为角色技能不同，看到的格子信息可以不同。

### 23.7 当前 Demo 的核心差距

当前 Demo 已有房间、角色、线索、结算、复盘，但与核心复刻仍有这些差距：

| 差距 | 当前 Demo | 核心复刻要求 |
|---|---|---|
| 标的结构 | 5 轮 5 个货柜 | 1 个仓库跑 5 轮 |
| 揭示方式 | 文本线索为主 | 格子仓库逐轮揭示 |
| 出价反馈 | 单轮拍卖结果较重 | 每轮反馈服务下一轮推价 |
| 模式重点 | 多种拍卖模式并列 | 先只打准明拍/暗拍 |
| 技能作用 | 有技能线索 | 技能必须改变仓库格子/估值视野 |
| UI重心 | 货柜卡片 + 线索栏 | 仓库格子 + 玩家状态 + 轮次反馈 |

### 23.8 开发优先级修正

新的开发优先级如下：

| 优先级 | 内容 | 说明 |
|---|---|---|
| P0 | 单仓运行时模型 | 一个仓库包含全部藏品、布局、真实价值、5轮揭示计划 |
| P0 | 五轮渐进揭示 | 每轮更新每名玩家的可见格子和估值信息 |
| P0 | 暗拍五轮逻辑 | 隐藏报价、轮后排名/差距反馈、第5轮成交 |
| P0 | 明拍五轮逻辑 | 公开当前价、领先者、放弃/跟价、第5轮成交 |
| P0 | 角色技能信息差 | 技能改变可见信息，并进入消息与复盘 |
| P0 | 格子仓库 UI | 右侧仓库布局随轮次和技能更新 |
| P0 | 每轮反馈 UI | 显示上一轮排名、差距、领先状态、风险提示 |
| P0 | 最终开箱与复盘 | 验证 5 轮判断是否正确 |
| P1 | 包装化仓型 | 普通仓、沉船仓、黑市仓等只作为不同背景和数值模板 |
| P2 | 市场/交易/商城 | 周边系统，当前不进入核心开发 |

### 23.9 包装补充原则

仓型可以补，但必须按包装处理：

- 普通旧仓、沉船仓、黑市仓、珠宝仓、宫廷仓都只是主题包装和数值分布差异。
- 每个仓型仍然遵守同一核心：单仓 5 轮、渐进揭示、明拍/暗拍、最终成交。
- 不因为包装引入额外复杂系统，例如门票、资产门槛、交易所、兑换商店。
- 包装可以影响初始描述、背景图、藏品类别、稀有度分布、陷阱概率。

### 23.10 本阶段不做的内容

为了保证核心 100% 复刻，本阶段明确不做：

- 市场交易。
- 商城。
- 活动仓入口。
- 局前装备系统。
- 多货币经济。
- 复杂角色养成。
- 付费外观。
- 道具商店。
- 排行榜和赛季。

这些内容不删概念，但不进入当前实现目标。当前目标只有一个：让玩家在同一个仓库的 5 轮明拍/暗拍中，真实地依靠信息、技能和对手行为推断最佳竞拍价格。
