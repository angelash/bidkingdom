# BidKing 源码实现逐模块还原度审计

审计时间：2026-05-23  
输入文档：`20260523_BidKing源码工程全模块整理.md`、`20260523_BidKing当前实现全模块整理.md`。  
口径：百分比是代码实现细节还原度的工程估计，综合数据接入、状态模型、流程、边界、UI 表现、测试证据。不是表行覆盖率，也不是玩家观感分。

## 总体结论

当前实现已经能运行 BidKing 风格的 Web 对局，并且接入了 52 张原配置表的大量数据；但它仍是 clean-room Web 复刻，不是 Unity 源码工程同构迁移。总体还原度按模块加权估计约 **59%**。

主要拉分项：

- `bidking-compat` 表数据覆盖高，52 张表都在项目中，核心字段可解析。
- 核心对局已具备五轮、明/暗拍可见性、成交线、自动技能、Bot、结算、亏损返利等关键体验。
- 服务端/前端有完整本地运行链和测试，能验证多数用户可见流程。

主要扣分项：

- 原服务端权威 Protobuf 模型没有同构，当前是自定义 `MatchRuntimeState`、REST/Socket.IO 和本地模拟。
- 实体仓库、物品 UID、格子容器、回收/送拍/陈列的统一生命周期仍未完全复刻。
- Unity UI/AssetBundle/动画/音频只能做视觉替代，不是像素级/资源级复刻。
- 任务、商店、交易、协会、排行等局外系统虽然有表驱动入口，但大量边界仍是简化状态机。

## 逐模块差距明细

### 1. 表数据与运行表类

当前完成：

- `packages/bidking-compat/src/tables` 已生成 52 张表的 TS 数据。
- `Constant / Condition / Reward / Item / Shop / Ticket / UIWnd / ErrorCode / Emoji / HeroSkin` 等有 helper 和测试。
- `match-core`、server、web 都已直接消费部分原表。

差距：

- 原源码有大量 `Table_*` 静态访问类和 `ConstantConfig`，当前是 TS 数组 + helper，不是运行时结构同构。
- 表字段接入深度不均：核心竞拍相关字段用得较多，商业/社交/系统表很多字段只展示或解释。
- `ExchangeRestock`、`ItemRestock` 等原 schema 缺字段的表仍有人工解释边界。

整改方向：

- 为每张表建立“字段 -> 行为入口 -> 测试”三列矩阵，区分 Display、Rule、State、Visual、Unused。
- 所有标记 Equivalent 的表，必须能指出至少一个源码方法和一个当前运行方法。

### 2. 网络协议与服务端权威

当前完成：

- `apps/server` 提供 REST + Socket.IO，本地房间可创建、准备、开局、出价、道具、表情、结算。
- `shared` 有统一快照类型，前端不直接读隐藏信息。

差距：

- 原源码围绕 `MsgHander` 和 Protobuf 协议号，当前没有 C2S/S2C 消息号同构。
- 原 `GameData`、`RoomData`、`StockContainerData` 是服务端权威对象，当前大部分状态由本地 TS 状态机模拟。
- 错误码、回包、通知时序和多模式分支没有逐协议覆盖。

整改方向：

- 建立协议覆盖矩阵：C2S/S2C 号、原 handler、当前 route/socket、缺失字段、是否状态权威。
- 对普通对局先实现 `S2C_45`、`S2C_119`、`C2S_34` 的兼容快照导出，再推进房间和模拟局。

### 3. 对局流程

当前完成：

- `submitBid` 已改为所有模式一轮只允许一次行动，出价 0 等价弃权。
- 明拍当前轮不再暴露出价，结算反馈后展示全员出价。
- `BidMap.auction_rounds_rate`、`RankMap`、`bid_fanli`、Hero/SkillEffect 均有运行入口。
- 最后一轮无人有效出价也能最终结算，避免无限推进。
- BattleItem 使用门禁已按源码普通局/私人房口径收敛为每名玩家每回合默认 1 次，私有快照给出本回合上限/已用/剩余次数。

差距：

- 原 `Battle_Handler` 是日志回放驱动，当前 `match-core` 是本地直接计算。
- 原“服务端下发每轮 `GameData`，客户端播放技能日志，再进入出价”没有完全同构。
- `GameSkillData.HitItemIndex` 已从误填仓库序号修正为命中数量；`TotalHitBoxIndex / AllHitItemAvgBoxIndex / AllHitItemAvgPrice / AllHitBoxAvgPrice / HitItemTotalPrice` 已与源码 `GetSkillParams` 的聚合参数消费口径一致。
- 源码 `StockBoxData.Position.X/Y` 是 `BoxId` 拆出的行/列；当前 `BidKingGameDataSnapshot.stockContainer` 已把前端 slot 坐标转换为源形 `position.x=row, position.y=column`。
- 高价提醒、保险关闭确认、上一轮多倍提醒、账户币和局内预算关系仍不完整。
- 私人房、模拟局、教程局三个分支的出价/结算协议仍简化。

整改方向：

- 让 `roundHistory` 首先产出源码形状的 `GameData/UserLog/SkillLog`，UI 再从该快照展示。
- 将出价前置校验拆为源码 UI 提醒层和服务端权威校验层。

## 还原度复评规则

以后每个模块不得只写“Verified”，必须同时给出：

- Source evidence：源码类/表/协议/资源入口。
- Current evidence：当前代码文件、函数、测试或截图。
- Behavior parity：状态模型、输入、输出、边界错误、时序是否一致。
- Data parity：原 ID、字段、行数、权重、概率、阈值是否一致。
- UI parity：布局、文本、交互、动画、音频是否一致或替代。
- Known gaps：明确列出缺口，不用“后续优化”这种空词。
