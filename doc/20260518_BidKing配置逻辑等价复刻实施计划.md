# 20260518 BidKing 配置逻辑等价复刻实施计划

> 目标：以本地授权分析结果为校验基准，完成《珍宝局》对 BidKing 核心竞拍体验的机制级等价复刻。`reverse/` 只保留本机分析镜像，不提交原始商业资源、原始表数据或反编译代码；可提交实现使用自有三国/古代包装、等价字段结构和等价数值曲线。

## 一、边界

允许：

- 使用本地 `reverse/bidking` 中的表结构、行数、字段语义和数值规律作为校验基准。
- 在项目内建立同名业务表模型，例如 `BidMap`、`Item`、`Drop`、`Hero`、`Skill`、`RankMap`、`RankAi`。
- 让运行配置在数量、规则、曲线、流程和交互节奏上对齐 BidKing。
- 替换所有显示名、文案、图标、模型、音频和视觉主题为《珍宝局》包装。

不做：

- 不提交原始 AssetBundle 图片、音频、模型、视频。
- 不提交反编译源码或原始商业表全文。
- 不做 DRM、联网绕过、作弊、付费解锁或资源再分发。

运行表原则：

- 表名、业务字段和模块边界按 BidKing 口径组织。
- 行数、成交线、轮次、掉落组、品质层级、占格形状、技能触发节奏做校验。
- 名称、描述、资源键、角色设定和 UI 文案全部换成三国/古代珍宝包装。
- 原始基准表只用于本地生成摘要和 parity 校验，不作为提交内容。

## 四、核心逻辑改造

新增表驱动运行模块：

```text
packages/match-core/src/bidking/
  bidMapResolver.ts
  dropResolver.ts
  warehouseBuilder.ts
  auctionRoundEngine.ts
  battleIntelligence.ts
  heroSkillEngine.ts
  rankAiEngine.ts
  settlementEngine.ts
```

改造目标：

- 开局从 `BidMap` 选择本局仓型。
- 用 `Drop + Item` 生成同一个仓库的全部藏品。
- 用 `Item.slot_type / grid_count` 生成右侧 Loot 格。
- 用 `BidMap.auction_rounds_rate` 驱动 5 轮成交线。
- 用 `RankMap` 驱动轮次时间、最低出价、竞拍类型和匹配参数。
- 用 `Hero + Skill + SkillGroup + SkillEffect` 驱动角色技能和情报揭示。
- 用 `RankAi` 替换当前简化 AI 估价参数。

## 五、流程目标

目标状态机：

```text
lobby
→ battle_prev_panel
→ battle_random
→ random_map_selected
→ auctioneer_intelligence_select
→ round_start
→ role_skill_reveal
→ battle_intelligence
→ bid_pop
→ round_bid_feedback
→ next_round 或 final_reveal
→ settlement
→ replay
```

当前 Web/Server 状态机可继续承载，但阶段命名、事件日志和 UI 组件要逐步贴近以上口径。
