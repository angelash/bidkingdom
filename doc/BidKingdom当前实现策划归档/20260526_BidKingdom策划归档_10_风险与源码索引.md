# 20260526 BidKingdom 策划归档 10 风险与源码索引

## 玩法还原验收原则

项目目标是 100% 还原原版《竞拍之王》的游戏性内容。任何数值、流程内核、技能链、技能目标、道具效果、掉落、AI、结算、任务、经济限制和局外落账差异，都应按原版源码和原始配置表修正。三国/古代化只允许作为文案、美术、音频和 UI 视觉包装存在。

若原版线上后端没有源码参考，则按协议推断实现，不作为阻塞项。推断优先级为：原始配置表和表类、Unity 客户端协议字段与 UI 消费行为、协议快照日志结构、可验证公式和边界行为、为补齐后端权威而写的最小等价逻辑。所有协议推断实现都要在报告中标注；证据增强时，用更精确的还原覆盖协议推断。

## 实现边界

| 验收项 | 规则 | 约束 |
| --- | --- | --- |
| 文档与源码一致性 | 共享类型、核心状态机、房间调度和前端派生状态只保留原版复刻流程 | 前端表现层不得接入非原版阶段或非原版流程 |
| 场景随机/情报动画 | 固定为 `intel` 内原版表现流程：匹配入口 `BidMap.map_group` 非空时 `openingCandidates` 驱动 `BattleRandom_Main`，存在地图公共情报时 `intelligenceChoices` 驱动 `IntelligencePanel` 四张暗牌 | 这是固定表现项，不是玩法阶段；重构不得因状态机收敛而删除动画，也不得让非随机拍场播放场景随机 |
| 源协议时间单位 | `GameData.ServerTime/NextRoundTime`、`SendAuctionData.SendTime`、`SendAuctionGameData.GameOverTime` 按原版客户端 `GetLocalTime()` 和 `TimerTool.ServerTime` 使用 Unix 秒 | 对外源协议快照不得写毫秒 |
| `Hero.packaged_name` 与原版姓名不同 | 运行用三国/古代包装名，原版实名见 20260525 竞买人归档 | 可保留为包装名，但 `Hero.id`、`cast_type`、职业/技能链、获取条件和数值含义必须逐项对齐原版 |
| 1136 行 Item 不宜复制进策划文档 | 本批文件只提炼分类和核心字段，完整行在 generated 表 | 若策划需要逐件校对，另导出 CSV/表格 |
| 系统类 SkillEffect | Category 15 为文本型价格位线索；16-28 按协议日志和 UI 行为推断 | 逐项按原版 `SkillEffect`、协议日志和对应 UI 表现补齐，不允许自定义效果替代 |
| 公共估值隐藏 | 原版 `Battle_Main.CalYuguPrice()` 的玩家侧估算来自已知格子，不是隐藏总估值；Bot 审计标 `protocol_inferred_hidden_range` | 玩家侧隐藏内部总估值，只展示已知情报估算 |
| Bot 行为树 | 读取 `RankAi`，服务端不可见估值和决策按协议推断实现 | 以原表 `RankAi` 字段语义和原版服务端/客户端可见行为为准；服务端不可见部分按协议推断实现并在审计字段标注 |
| 成交线解释需锁定 | 实现按领先差距判断，等价于按 `BidMap.auction_rounds_rate` 判断最高价与第二价关系 | 以原版 `BidMap.auction_rounds_rate` 和可验证行为统一验收，测试覆盖每轮成交、未成交、平价和最终轮 |
| 字段级协议映射 | `GameData`、`RoomData`、`SendAuctionGameData`、`MailItemData`、`ShopStatusData` 已落到共享类型、运行时和服务端文件 | 修改这些协议字段时同步核对 `20260527_BidKingdom策划归档_11_源码比对细化补充.md` |
