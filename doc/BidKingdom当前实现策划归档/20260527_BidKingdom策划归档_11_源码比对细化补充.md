## 表现流程固定点

| 表现 | 触发 | 数据 | 时长 | 实现 |
| --- | --- | --- | ---: | --- |
| 普通匹配 | 战前点击开始行动 | `RoomSnapshot` 内部上下文、`selectedBidMapId`、预计匹配秒数 | 匹配成功前持续 | `matchGame()` 创建内部上下文；`BattlePrevPanelView` 收起场景详情并在大场景地图显示匹配悬浮窗；成功后进入随机拍场或战斗 |
| 场景随机 | 匹配入口 `BidMap.map_group` 非空且首轮 `intel` 开始 | `currentRound.openingCandidates`，最后一项为选中拍场 | 1600 ms | `roundPresentationOverlay()` 返回 `battle_random`，`BattleRandomOverlay` 渲染候选和结果 |
| 情报暗牌 | 普通局首轮 `intel` 开始，存在地图公共情报 | `currentRound.intelligenceChoices`、`currentRound.intelligenceClue` | 4600 ms | `roundPresentationOverlay()` 返回 `intelligence_panel`，`IntelligencePanelOverlay` 先显示四张暗牌，约 1600 ms 后明牌，明牌后停留 3000 ms；第二轮起不再生成或播放暗牌情报 |
| 中心情报列表 | 暗牌表现完成后，或后续回合/试宝令触发时 | `currentRound.skillFeed` 累计可见条目 | 逐条串行 | `MarketIntelPanel` 常驻渲染已出现条目；仅本轮新增条目按 `map -> hero -> item -> manual -> auto` 排序并播放提示卡，卡片隐藏/移动完成后该行进入列表 |
| 终局揭露 | `settleCurrentRound()` 后进入最终 `reveal` | `currentRound.warehouseSlots`、`currentRound.revealedItems` | 按品质 1000/1000/1000/2000/2000/3000 ms | `revealDelayForItem()` 调度逐件揭露，`BattleFinalCeremony` 用 `img_loading_1` 搜索态等待，结束后同帧显示品质和藏品 |
| 操作窗口 | `intel` 和 `auction` | `canUseBattleItem`、`canBid` | 由服务器阶段控制 | 开局动画和技能提示由 `intel` 阶段窗口承载；完成后进入出价窗口，不新增第二套玩法状态 |

`intel` 阶段时长按单个玩家可见的最大日志数计算，不把其他玩家不可见的私有角色技能计入等待。首轮时长 = 场景随机表现 + 情报暗牌表现 + 可见技能提示入列时长；后续回合只计算可见技能提示入列时长，并保留 3200 ms 下限。

## 错误边界

| 场景 | 实现行为 |
| --- | --- |
| `coreMode=false` | `createMatch()` 抛出 `Only BidKing core mode is supported` |
| 玩家数不等于 `BidMap.bidder_number` | `createMatch()` 抛出人数错误 |
| `BidMap`、随机子拍场、开场候选、`RankMap`、`Drop`、`Item` 引用缺失 | 对应运行时抛错 |
| `Hero`、`Skill`、`SkillEffect`、`BattleItem` 源链缺失 | 技能或试宝令运行时抛错 |
| `RankAi` 行缺失或权重区间为空 | Bot 决策运行时抛错 |
| 回合缺 `auctionDurationMs` | `match.ts` 和 `roomRoundRuntime.ts` 抛错 |
| 邮件/商店/送拍配置不存在或条件不足 | 对应服务端运行时返回错误，不生成替代落账 |

## 核对结论

本轮补充后，策划文档对局内协议快照、房间协议、表现动画、技能/道具日志、RankAi、送拍、邮件、商店、UIWnd 和系统表的实现落点都有明确索引。后续新增或重构这些模块时，按本分册的“协议字段 -> 实现函数 -> 固定口径”核对，不引入第二套玩法规则。
